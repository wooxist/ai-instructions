# Task 3: 다양한 도메인 예시 추가

## 목적
3.4절 마지막에 새로운 섹션을 추가하여 콘텐츠 작성, 코드 리뷰, 고객 지원 등 3가지 도메인의 Before/After 예시를 제공하여 전이 학습을 지원합니다.

## 배경
현재 3.4절은 "주간 판매 보고서" 단일 예시만 제공하여, 데이터 분석 외 다른 도메인의 독자는 자신의 업무에 적용하기 어렵습니다. 다양한 도메인의 간단한 예시를 추가하여 독자가 패턴을 발견하고 자신의 업무에 전이할 수 있도록 합니다.

## 입력
- `/doc/book1/vol-1-part-1-chapter-03.md` (현재 버전)
- 개선안: 이전 대화의 "개선 제안 #5" 내용

## 작업 내용

### 1. 삽입 위치 확인
3.4절의 "Before와 After의 실질적 차이" 섹션 마지막 다음에 새로운 섹션을 추가합니다.

### 2. 추가할 내용
다음 내용을 추가합니다:

```markdown
### 다른 도메인의 간단한 예시

주간 판매 보고서 외에도 인스트럭션은 다양한 분야에 적용됩니다. 
여러분의 업무와 가까운 예시를 찾아보세요.

#### 예시 1: 콘텐츠 작성 (마케팅/블로그)

**Before (채팅창)**:
```
"블로그 글 써줘"
```

**After (핵심 요소만)**:
```markdown
📄 seo-blog-post.instruction.md

## 1. 목적 ⭐
SEO 최적화된 블로그 포스트로 자연 유입 트래픽 증대

## 2. 역할 ⭐
10년차 콘텐츠 마케터 (SEO와 독자 참여 최적화 전문)

## 3. 입력 ⭐
- 주제: 메인 키워드 1개
- 보조 키워드: 3-5개
- 타겟 독자: 페르소나 설명

## 4. 처리 방법 ⭐
1. 키워드 밀도 계산 (메인 키워드 1-2%)
2. 제목에 메인 키워드 포함 (60자 이내)
3. 본문 작성 (1000-1500자)
4. 메타 설명 작성 (150-160자)

## 5. 출력 ⭐
- 제목: 60자 이내, 메인 키워드 포함
- 본문: 1000-1500자, 마크다운 형식
- 메타 설명: 150-160자
- 구조: 도입(150자) / 본문(800자) / 결론(150자)

## 6. 제약 조건 🔵
- 키워드 스터핑 금지 (부자연스러운 키워드 반복)
- 클릭베이트 제목 금지
- 정확하지 않은 통계 인용 금지
```

**시스템 속성 검증**:
- ✅ 재사용: 주제/키워드만 바꿔서 매주 사용
- ✅ 일관성: SEO 기준을 항상 충족
- ✅ 확장: 다양한 주제에 적용 가능

#### 예시 2: 코드 리뷰 (개발)

**Before (채팅창)**:
```
"이 코드 리뷰해줘"
```

**After (핵심 요소만)**:
```markdown
📄 code-review-security.instruction.md

## 1. 목적 ⭐
Pull Request의 보안 취약점과 성능 이슈 조기 발견

## 2. 역할 ⭐
시니어 보안 엔지니어 (OWASP Top 10 전문)

## 3. 입력 ⭐
- diff 파일 또는 PR URL
- 언어/프레임워크 정보
- 배포 환경 (production/staging)

## 4. 처리 방법 ⭐
1. OWASP Top 10 체크리스트 검증
2. 인증/인가 로직 검토
3. SQL Injection 가능성 검사
4. XSS 취약점 스캔
5. 민감 정보 하드코딩 확인

## 5. 출력 ⭐
- 형식: 마크다운 표
- 컬럼: 심각도 | 위치 | 문제 | 권장사항
- 심각도: Critical/High/Medium/Low
- 우선순위: Critical/High부터 정렬

## 6. 제약 조건 🔵
- 코드를 직접 수정하지 말고 제안만 하기
- 추측이 아닌 명확한 근거 제시
- False Positive 최소화 (확실한 것만 보고)
```

**시스템 속성 검증**:
- ✅ 재사용: 모든 PR에 자동 적용 가능
- ✅ 일관성: 보안 기준이 항상 동일
- ✅ 협업: 팀 전체가 같은 보안 기준 공유

#### 예시 3: 고객 지원 (CS)

**Before (채팅창)**:
```
"FAQ 답변 만들어줘"
```

**After (핵심 요소만)**:
```markdown
📄 faq-answer-generator.instruction.md

## 1. 목적 ⭐
고객 문의에 대한 일관되고 정확한 FAQ 답변 생성

## 2. 역할 ⭐
5년차 고객 지원 전문가 (공감과 명확성 중시)

## 3. 입력 ⭐
- 고객 질문: 원문 그대로
- 제품 카테고리: 결제/배송/반품/기술지원
- 긴급도: 일반/긴급

## 4. 처리 방법 ⭐
1. 질문 의도 파악 (무엇을 해결하고 싶은가?)
2. 관련 정책 확인 (약관/가이드 참조)
3. 단계별 해결 방법 작성
4. 추가 문의 경로 안내

## 5. 출력 ⭐
- 길이: 200자 이내
- 구조: 공감(1줄) / 해결책(3-4줄) / 추가 안내(1줄)
- 톤: 친절하고 전문적
- 형식: 마크다운 (불릿 리스트 가능)

## 6. 제약 조건 🔵
- 확인되지 않은 정보 제공 금지
- 책임 회피성 답변 금지 ("아마도", "~인 것 같습니다")
- 고객 개인정보 언급 금지
```

**시스템 속성 검증**:
- ✅ 재사용: 유사 질문에 즉시 적용
- ✅ 일관성: CS 팀 전체가 같은 톤 유지
- ✅ 확장: 신규 제품/서비스 추가 시 확장 용이

---

### 여러분의 도메인에도 적용해보세요

위 예시들의 공통 패턴을 보셨나요?

1. **Before는 항상 모호함**: "~해줘"
2. **After는 항상 구체적**: 목적, 역할, 입출력이 명확
3. **시스템 속성이 만족됨**: 재사용/일관성/확장성 확보

이 패턴을 여러분의 업무에 적용하면:
- 데이터 분석 → 주간 보고서 / 대시보드 생성 / 이상 탐지
- 디자인 → 와이어프레임 피드백 / 컬러 팔레트 제안 / 레이아웃 검토
- 영업 → 제안서 작성 / 미팅 요약 / 후속 이메일
- 교육 → 강의 자료 생성 / 퀴즈 문제 출제 / 피드백 작성

3장 끝의 실습 과제에서 여러분의 도메인으로 직접 만들어보세요!
```

### 3. 검증 사항
- [ ] 3개 도메인이 서로 다른 분야를 대표하는가
- [ ] 각 예시가 필수 5가지 + 권장 1개를 포함하는가
- [ ] Before/After 대비가 명확한가
- [ ] 독자 친화적 어투를 유지하는가
- [ ] 추가 내용이 약 1500자인가

## 출력
수정된 3.4절이 포함된 Chapter 3 파일

## 완료 기준
- [ ] 3개 도메인 예시가 추가됨
- [ ] 공통 패턴 설명이 포함됨
- [ ] 기존 내용과 자연스럽게 연결됨
- [ ] 독자 친화적 문체 유지
- [ ] 분량이 약 1500자 추가됨

## 실행 방법
1. Chapter 3 파일을 연다
2. 3.4절의 마지막 부분을 찾는다
3. Filesystem:edit_file을 사용하여 새로운 섹션을 추가한다
4. 수정된 파일을 확인한다
