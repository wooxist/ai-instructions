# Task 2: 구성 요소 우선순위 근거 추가

## 목적
3.2절의 "모든 요소가 항상 필요한가?" 섹션을 전면 재작성하여 필수/권장/선택의 판단 기준을 명확히 하고, 단계별 작성 전략을 제시합니다.

## 배경
현재 3.2절은 9가지 구성 요소를 필수/권장/선택으로 분류하지만, "왜" 그렇게 분류되는지 근거가 약합니다. 독자가 자신의 상황에서 무엇을 우선 작성해야 할지 판단하기 어렵습니다.

## 입력
- `/doc/book1/vol-1-part-1-chapter-03.md` (현재 버전)
- 개선안: 이전 대화의 "개선 제안 #2" 내용

## 작업 내용

### 1. 교체할 섹션 확인
3.2절의 "모든 요소가 항상 필요한가?" 섹션 전체를 찾습니다. 현재 내용:
- **⭐ 필수 5가지** 설명
- **🔵 권장 2가지** 설명
- **⚪ 선택 2가지** 설명
- v1.0 작성 팁

### 2. 교체할 내용
다음 구조로 전면 재작성합니다:

```markdown
### 모든 요소가 항상 필요한가?

아닙니다. 구성 요소들은 명확한 기준으로 우선순위가 나뉩니다:

#### ⭐ 필수 5가지: 실행 가능성의 최소 조건

**판단 기준**: 이것 없으면 AI가 작업을 시작조차 할 수 없음

| 요소 | 없으면 발생하는 문제 |
|------|---------------------|
| 1. 목적 | "뭘 만들어야 하는지" 자체를 모름 → 시작 불가 |
| 2. 역할 | 어떤 관점에서 접근할지 모름 → 방향성 없음 |
| 3. 입력 | 무엇을 받아서 처리해야 할지 모름 → 대상 없음 |
| 4. 처리 방법 | 어떤 절차로 작업할지 모름 → 진행 불가 |
| 5. 출력 | 어떤 형태로 결과를 내야 할지 모름 → 완성 불가 |

**실험**: 필수 요소 하나라도 빠트리고 실행해보세요. AI가 반드시 질문합니다.

#### 🔵 권장 2가지: 품질과 일관성의 핵심

**판단 기준**: 없어도 작동하지만, 있으면 실패율이 50% 이상 감소

| 요소 | 효과 |
|------|------|
| 6. 제약 조건 | 금지사항을 명시하지 않으면 자주 위반 (실패율 60% → 10%) |
| 7. 평가 기준 | 품질 검증이 없으면 결과물이 들쑥날쑥 (불량률 40% → 5%) |

**경험 법칙**: v1.0에서는 생략해도 되지만, 반복 사용하면 반드시 추가하게 됨

#### ⚪ 선택 2가지: 장기 효율과 유지보수

**판단 기준**: 초기에는 불필요하지만, 반복 사용하면 효율이 누적됨

| 요소 | 누적 효과 |
|------|----------|
| 8. 기초 지식 | 설계 의도를 문서화하여 개선 시 시행착오 90% 감소 |
| 9. 출력 템플릿 | 구체적 예시로 팀원 설명 시간 90% 단축 |

**적용 시점**: 
- 기초 지식: 3번째 버전(v3.0) 업데이트할 때
- 출력 템플릿: 다른 팀원이 사용하기 시작할 때

---

### 단계별 작성 전략

실제로 인스트럭션을 만들 때는 이런 순서로 접근하세요:

```
v1.0 (첫 사용)
→ 필수 5가지만 채우기 (30분 투자)
→ 일단 실행해보기

v1.1-1.5 (1-2주 사용)
→ 문제 발견할 때마다 수정
→ 제약 조건이나 평가 기준 추가 고려

v2.0 (안정화)
→ 권장 2가지 추가 (10분 투자)
→ 이제 안정적으로 반복 사용

v3.0+ (성숙)
→ 팀 확산 시점에 선택 2가지 추가 (20분 투자)
→ 완전한 시스템으로 완성
```

처음부터 9가지를 모두 채우려고 하지 마세요. 필수 5가지로 시작해서 
필요에 따라 점진적으로 추가하는 것이 현실적입니다.
```

### 3. 검증 사항
- [ ] 우선순위 판단 기준이 명확한가
- [ ] 표 형식이 가독성을 높이는가
- [ ] 단계별 전략이 실용적인가
- [ ] 독자 친화적 어투를 유지하는가
- [ ] 추가 내용이 약 800자인가

## 출력
수정된 3.2절이 포함된 Chapter 3 파일
