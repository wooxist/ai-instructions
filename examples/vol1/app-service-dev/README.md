# 앱 서비스 개발 통합 예제

**목적**: AI 협업 가이드북 1권 (Part 1-4, 1-10장)의 학습 내용을 하나의 프로젝트로 통합

**예제 시나리오**: 간단한 할일 관리 앱 "TaskFlow" 개발

---

## 🎯 학습 목표

이 통합 예제를 통해:
1. **Part 1-4의 진화 과정**을 실제 프로젝트로 경험
2. **각 장의 학습 포인트**를 실무 맥락에서 적용
3. **단일 프롬프트 → 조직 아키텍처**까지 자연스러운 성장 경로 이해

---

## 📱 TaskFlow 앱 소개

**개념**: 개인/팀용 할일 관리 앱
- 할일 등록/수정/완료
- 우선순위 설정
- 마감일 알림
- 진행률 대시보드
- 간단한 팀 협업 기능

**왜 TaskFlow인가?**
- ✅ 모든 기능 조직이 자연스럽게 참여 (기획/전략/디자인/개발/QA/마케팅)
- ✅ 복잡도 조절 가능 (Part별로 기능 확장)
- ✅ 실무에서 흔한 유형
- ✅ 구체적이면서도 범용적

---

## 🚀 Part별 진화 과정

### Part 1: 단일 프롬프트 단계 (1-4장)
**디렉토리**: `part1-prompts/`

**시나리오**: AI에게 한 번의 대화로 작업 완료
- "TaskFlow 앱 기획서를 작성해줘"
- "간단한 화면 구성을 설명해줘"
- "필요한 기능 목록을 정리해줘"

**학습 포인트**:
- 1장: 좋은 프롬프트 작성 (명확한 요청, 제약사항, 출력 형식)
- 2장: 질문 유형별 활용 (폐쇄형/개방형/비교형)
- 3장: 8가지 구성 요소를 포함한 인스트럭션
- 4장: SSOT, SoC 메타 원칙 적용

**한계**:
- 복잡한 작업은 한 번에 처리 어려움
- 일관성 유지 어려움
- 재사용성 낮음

---

### Part 2: 단일 에이전트 (5-7장)
**디렉토리**: `part2-single-agent/`

**시나리오**: 하나의 전문 에이전트가 여러 역할 수행
- "기획자 에이전트": 시장 조사 → 요구사항 정의 → 기획서 작성
- 역할/제약/입출력이 명확히 정의된 구조화된 접근

**학습 포인트**:
- 5장: 에이전트 설계 (역할, 제약, 안전장치)
- 6장: 입출력 명세 (JSON Schema, 샘플)
- 7장: 워크플로우 패턴 (순차/병렬/생성-검증)

**개선**:
- 재사용 가능한 에이전트
- 명확한 입출력 인터페이스
- 단계별 워크플로우

**한계**:
- 한 에이전트가 모든 것을 처리 (과부하)
- 전문성 한계
- 병렬 처리 불가

---

### Part 3: 다중 에이전트 파이프라인 (7.5-9장)
**디렉토리**: `part3-pipeline/`

**시나리오**: 전문 에이전트들이 파이프라인으로 협업
- 기획자 → 디자이너 → 개발자 → QA 엔지니어
- 각 에이전트가 전문 분야 집중
- 산출물이 다음 단계로 전달

**학습 포인트**:
- 7.5장: 상태 관리 (status.json, 디렉토리 구조, 재시도 정책)
- 8장: 성능 최적화 (모델 선택, 토큰 절감)
- 9장: 평가와 검증 (A/B 테스트, 오답 노트)

**개선**:
- 전문성 향상 (각자 역할 집중)
- 병렬 처리 가능
- 상태 추적 및 오류 처리

**한계**:
- 의사결정 주체 불명확
- 우선순위 조정 어려움
- 예외 상황 대응 수동

---

### Part 4: 조직 아키텍처 (10장)
**디렉토리**: `part4-organization/`

**시나리오**: PM 아키텍트 + 전문 워커 조직
- **PM 아키텍트**: 전체 조율, 의사결정, 우선순위 관리
- **워커들**:
  - 기획자 (Product Manager)
  - 전략가 (Strategist)
  - 디자이너 (UI/UX Designer)
  - 개발자 (Frontend/Backend Developer)
  - QA 엔지니어 (Quality Assurance)
  - 마케터 (Marketing Specialist)

**학습 포인트**:
- 10장: 아키텍트 패턴 (의사결정 체계, 역할 분리)
- 조직도 설계
- 산출물 인터페이스 정의
- 승인/반려 라우팅

**최종 형태**:
- 명확한 의사결정 체계
- 전문 워커들의 협업
- 확장 가능한 구조
- 실패 처리 및 피드백 루프

## 🎓 학습 가이드

### 순서대로 진행
1. **Part 1부터 시작**: 가장 간단한 프롬프트 방식
2. **각 Part의 한계 체감**: 왜 다음 단계가 필요한지 이해
3. **점진적 개선**: 이전 단계의 문제를 다음 단계에서 해결
4. **비교 분석**: Part별 차이점, 장단점 정리

### 실습 체크리스트
각 Part 디렉토리의 `checklist.md`를 참고하여:
- [ ] 예제 코드/템플릿 실행
- [ ] 자신의 프로젝트로 변형
- [ ] 한계점 경험 및 기록
- [ ] 다음 Part와 비교
