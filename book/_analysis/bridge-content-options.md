## 최종 결정

### ✅ 선택: 옵션 1 - 7.5장 신규 추가

**근거:**
1. **학습자 경험 우선**: 갭 분석에서 확인된 치명적 난이도 갭을 해소하려면, 명확하고 독립적인 중간 단계가 필수적입니다.
2. **장기적 가치**: Phase 3, 4에서 추가 고급 주제를 다룰 때, .5장 체계를 활용한 유연한 확장이 가능합니다.
3. **데이터 기반 개선**: 7.5장에 대한 독립적인 피드백 수집으로 지속적 개선이 가능합니다.
4. **유지보수성**: 독립 파일로 관리하면 버전 관리, 협업, 리팩토링이 훨씬 수월합니다.

**트레이드오프 수용:**
- Part 3가 4개 장이 되는 것은 큰 문제가 아닙니다. 실제로 Part 2도 3개 장(4-6장)이고, Part 4도 3개 장(10-11장 + 11장 하위)이므로, 일관성이 크게 깨지지 않습니다.
- 파일 하나 추가하는 기술적 오버헤드는 학습 경험 개선 효과에 비해 미미합니다.

## 7.5.1 복잡도의 진화: 언제 더 많은 에이전트가 필요한가?
- 단순 협업(2개) vs 중간 협업(3-4개) vs 복잡 협업(6개+)
- 의사결정 기준: 작업 분리 가능성, 전문성 요구, 병렬 처리 필요성

## 7.5.2 3-4개 에이전트 협업 패턴
- **예시: 콘텐츠 팀 워크플로우**
  - 작성 에이전트 → 검토 에이전트 → 편집 에이전트 → 발행 에이전트
- 역할과 책임의 명확한 분리
- 파이프라인 + 생성-검증 하이브리드 패턴

## 7.5.3 파일 기반 상태 관리 입문
- 왜 상태 추적이 필요한가? (작업 진행 상황, 에러 복구)
- 간단한 상태 파일: `status.json`
  ```json
  {
    "current_step": "review",
    "status": "in_progress",
    "timestamp": "2025-10-06T10:00:00Z"
  }
  ```
- 에이전트가 상태를 읽고 쓰는 방법

## 7.5.4 기본 디렉토리 구조 설계
- 단일 디렉토리의 한계 (파일 10개 이상 시 혼란)
- 작업별 폴더 분리: `/tasks/task-01/`, `/tasks/task-02/`
- 공통 자원 폴더: `/shared/templates/`
- 산출물 폴더: `/outputs/`

## 7.5.5 산출물 표준화: 텍스트에서 JSON으로
- 왜 구조화된 데이터가 필요한가?
- 간단한 JSON 산출물 예시
  ```json
  {
    "task_id": "content-001",
    "status": "completed",
    "output": {
      "title": "...",
      "content": "..."
    }
  }
  ```
- 다음 에이전트가 JSON을 읽는 방법

## 7.5.6 에러 처리와 재시도 전략
- 에이전트 실패 시나리오 (타임아웃, 형식 오류, 품질 불합격)
- 간단한 재시도 로직 (workflow.yaml에 `max_retries: 3`)
- 실패 로그 기록: `error.log`

## 7.5.7 다음 단계로: 10장 준비하기
- 지금까지 배운 것: 3-4개 에이전트, 기본 상태 관리, 간단한 디렉토리 구조
- 10장에서 배울 것: 계층적 조직(메타/아키텍트/워커), 복잡한 디렉토리 구조, Job 관리
