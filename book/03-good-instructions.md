# 3장. 좋은 인스트럭션

**1부: 인스트럭션의 기초와 설계 원칙**

**목적:** 명령문 자체의 품질을 높이는 법을 배운다.

### 이 장에서 배우는 것
- 좋은 지시를 만드는 5가지 원칙(명확성·구체성·단계성·제약·안티패턴)
- 원칙별 바로 쓰는 문장 템플릿과 나쁜예 대조
- 검증 가능한 성공 기준으로 마무리하는 방법

## 인스트럭션이란? (개요)

인스트럭션(Instruction)은 체계화된 프롬프트로, 단순한 입력문을 넘어 일관성과 협업성을 갖춘 지시 체계를 의미합니다. 이는 AI가 특정 작업을 수행할 때 매 실행마다 참조하는 기초 지식과 행동 지침의 역할을 하며, 재사용 가능하고 표준화된 형태의 지시 방식입니다.


### 인스트럭션 설계의 이점

- 결과 예측성: 동일 인스트럭션 반복 사용 시 일관된 결과
- 사용자 독립성: 사용자에 따른 품질 편차 감소
- 효율성: 재사용을 통한 작성 시간·노력 절감
- 품질 향상: 테스트·개선을 통한 지속 최적화
- 협업 용이성: 공유·표준화를 통한 팀 자산화

### 설계가 필요한 상황

- 반복적 작업: 보고서, 코드 리뷰, 데이터 분석 등
- 팀 협업: 고객 응대, 콘텐츠 제작, 데이터 처리 등
- 품질 중요: 공식 문서, 대외 커뮤니케이션, 제품 설명 등
- 지속적 개선: 요구사항 변화·신규 AI 기능 반영 필요 시
- 표준화 필요: 규정 준수, 지식 전수, 프로세스 개선

### 인스트럭션의 일반적 구성 요소

- 목적 설명: 인스트럭션의 목적과 용도 명시
  "이 인스트럭션의 목적은 주간 판매 데이터를 분석하여 핵심 추세와 인사이트를 도출하는 것입니다."

- 역할 정의: AI가 수행해야 할 역할 구체화
  "당신은 데이터 분석가로서, 판매 데이터를 분석하고 경영진을 위한 인사이트를 도출해야 합니다."

- 입력 사양: 처리 대상 정보의 형식·구조 설명
  "입력 데이터는 CSV 형식이며, 날짜, 제품 ID, 판매량, 판매액, 지역 정보를 포함합니다."

- 처리 방법: 단계별 처리 절차 지시
  "1. 제품별 총 판매량과 판매액을 계산하세요.
   2. 지역별 판매 추세를 분석하세요.
   3. 전주 대비 증감률을 계산하세요.
   4. 상위 5개 제품과 하위 5개 제품을 식별하세요."

- 출력 요구사항: 기대하는 결과물의 형식·내용 명시
  "분석 결과는 다음 섹션으로 구성된 마크다운 형식의 보고서로 작성하세요:
   1. 요약(주요 발견 3-5개)
   2. 제품별 판매 분석(차트 포함)
   3. 지역별 판매 분석(표 포함)
   4. 전주 대비 성과 비교
   5. 권장 조치사항"

- 제약 조건: 한계·조건 명시
  "보고서는 500단어를 초과하지 않아야 하며, 기술적 용어 사용을 최소화하고, 시각적 요소를 포함해야 합니다."

- 평가 기준: 품질 측정 방법 제시
  "보고서는 다음 기준으로 평가됩니다: 데이터 분석의 정확성, 인사이트의 실용성, 시각적 요소의 명확성, 권장사항의 구체성"

- 기초 지식 및 원칙: 기본적으로 참조할 원칙 정의
  "분석 시 다음 원칙을 항상 준수하세요:
   1. 데이터 무결성: 결측값과 이상치를 항상 고려하고 처리합니다.
   2. 인과관계 vs 상관관계: 단순 상관관계를 인과관계로 해석하지 않습니다.
   3. 비즈니스 중심: 모든 분석은 비즈니스 가치와 연결되어야 합니다."

### 좋은 인스트럭션의 핵심 특성

1. 명확성: 모호함 없이 정확한 지시 제공
2. 구체성: 기대치와 세부 요구사항을 구체적으로 명시
3. 구조화: 일관된 형식과 구조로 이해·따르기 쉽게 설계
4. 재사용성: 유사 작업에 반복 활용 가능
5. 검증 가능성: 평가 기준을 포함해 객관적 성능 측정 가능
6. 참조성: 실행 과정에서 지속 참조되는 기초 지식 역할

## 3.1 명확성 (Clear)
명확성은 모호함 없이 의도와 요구를 전달하는 것입니다.

### 언제 유용한가
오해로 비용이 커지는 업무(보고·요청·검토·승인) 전반

### 언제 안 쓰나
아이디어 발산 단계에서 즉흥적 브레인스토밍 자체를 목표로 할 때(탐색형이 우선)

### 예시
"고객 불만 상위 3개 유형을 요약하고, 각 유형별 원인 1줄과 개선안 1개를 불릿으로 정리해줘."

- (개발) "지난 24시간 오류 로그에서 상위 3개 에러코드와 발생 횟수를 표로 정리(코드·횟수·최근 발생시각), 10줄 이내"
- (마케팅) "지난주 캠페인 A의 클릭률/전환율/CPA를 불릿 3개로 요약, 마지막에 개선안 1개"
- (운영) "지난달 불만 상위 3유형과 원인(1줄)·대응 권장안(1개) 불릿"

### 나쁜예
"고객 불만 좀 알려줘"

- (개발) "로그 좀 정리해줘"
- (마케팅) "캠페인 성과 알려줘"
- (운영) "불만 좀"

### 이렇게 적어보세요
"대상: 지난달 고객 불만. 형식: 불릿 3개. 각 불릿 = 유형·원인(1줄)·개선안(1개). 길이: 총 120자 내"

### 자주하는 실수
‘적절히/충분히/빠르게’ 같은 모호어 사용, 독자·형식·길이 미지정

### 더 좋게
수락 기준을 함께 적기(예: 불릿 3개·중복 없음·행동가능한 개선안)

### 성공 기준 예시
불릿 3개, 중복 없음, 각 불릿에 행동가능한 개선안 1개 포함

참고: 형식/컨텍스트는 [6장. 입력과 출력 설계](06-input-output.md), 제한/대응 전략은 [5장. 역할(Agent)과 제약(Constraint) 설계](05-agent-constraints.md)에서 자세히 다룹니다.

## 3.2 구체성 (Specific)
구체성은 범위·대상·형식을 분명히 적어 결과의 산포를 줄입니다.

### 언제 유용한가
요약/정리/보고 같은 산출물 품질 편차를 줄일 때

### 언제 안 쓰나
문제 정의 전 탐색 단계(먼저 질문 정교화 필요)

### 예시
"제품 A 유입 경로별 전환율을 지난 8주 주차 기준으로 표 1개로 정리"

- (개발) "API `/v1/search` 95퍼센타일 응답시간을 최근 7일, 일별·시간대별 라인차트 1개로"
- (마케팅) "제품 B 유입경로별 전환율을 최근 8주 주차 기준 표 1개(경로·전환율·증감)"
- (운영) "SKU 상위 10개 재고/소진속도/재입고예정일을 표(컬럼 3개)로 요약"

### 나쁜예
"전환율 분석해줘"

- (개발) "성능 분석해줘"
- (마케팅) "전환율 보고서 만들어"
- (운영) "재고 상황 알려줘"

### 이렇게 적어보세요
"대상: 제품 A, 기간: 최근 8주, 단위: 주차, 형식: 표 1개(경로·전환율·추이)"

### 자주하는 실수
대상/기간/단위 누락, 형식·길이 미정, 평가 기준 없음

### 더 좋게
예시 1줄을 함께 제시해 기대 형식 정렬

### 성공 기준 예시
표 1개(경로·전환율·추이), 기간·단위 조건 충족, 누락 없음

참고: 형식/컨텍스트는 [6장. 입력과 출력 설계](06-input-output.md), 제한/대응 전략은 [5장. 역할(Agent)과 제약(Constraint) 설계](05-agent-constraints.md)에서 자세히 다룹니다.

## 3.3 단계성 (Step-wise)
복잡한 일을 작은 단계로 쪼개 실행/검증을 쉽게 합니다.

### 언제 유용한가
리뷰/개선/생성·검증이 반복되는 업무(코드/문서/데이터 처리)

### 언제 안 쓰나
단일 액션으로 끝나는 단순 과업(과도한 분해는 오히려 비효율)

### 예시
"1) 요구 정리 2) 초안 생성 3) 자기검토 체크리스트 4) 최종본 출력"

- (개발) "1) 복잡도 높은 함수 3개 식별 2) 테스트 보강 3) 함수 분리 4) 성능 비교 리포트(표 1개)"
- (마케팅) "1) 히트맵 분석 요약 2) 이탈구간 가설 3) A/B 후보 3개 4) 실험 설계(지표·기간·표본)"
- (운영) "1) 현행 프로세스 다이어그램 2) 병목 3곳 정의 3) 개선안 3개 4) 기대효과·리스크 표"

### 나쁜예
"완성본 바로 만들어줘"

- (개발) "리팩토링 해줘"
- (마케팅) "랜딩 페이지 개선"
- (운영) "온보딩 개선해줘"

### 이렇게 적어보세요
"아래 단계를 순서대로 수행하고, 각 단계 종료 시 산출물을 보여줘: 1)… 2)… 3)… 4)…"

### 자주하는 실수
단계 간 수락 기준/출력이 없음, 단계가 누락·중복

### 더 좋게
각 단계에 ‘성공 기준’과 ‘다음 단계로 넘길 입력’을 명시

### 성공 기준 예시
각 단계 산출물 제출, 체크리스트 100% 충족, 최종 성능/품질 개선 수치 명시

참고: 처리 방법의 상세 내용은 [7장. 처리 방법과 워크플로우 설계](07-process-workflow.md)에서 자세히 다룹니다.

## 3.4 제약 조건과 한계 설정
제약은 현실 세계의 조건(시간·환경·정책·품질선)을 미리 적는 것입니다.

### 언제 유용한가
실행 가능성·법적/보안 준수·시간 박스가 중요한 과업

### 언제 안 쓰나
탐색형 초기 단계에서 아이디어 폭을 넓히고자 할 때(일시적으로 제약 완화)

### 예시
"환경: MacOS·Python 3.10, 시간: 2h, 제약: 인터넷 제한·외부 API 금지, 성공 기준: 오류<1%"

- (개발) "환경: Linux·Py3.10, 시간: 2h, 제약: 외부 API 금지·오프라인, 목표: 오류<1%, 출력: 수정 코드·테스트 리포트"
- (마케팅) "예산: 300만원, 채널: 인스타/유튜브, 톤: 친근·신뢰, 금지: 과장표현, 성공기준: CTR>2%, CPA<2만원"
- (운영) "형식: 마크다운 템플릿 준수, 분량: 2p, 범위: 개인정보 취급만, 준거: 내부 규정 v3·PIPA 조항"

### 나쁜예
"최대한 빨리 잘 만들어줘"

- (개발) "최대한 빨리 고쳐"
- (마케팅) "바이럴 되게 만들어"
- (운영) "정책 문서 빨리"

### 이렇게 적어보세요
"환경/버전/제약/목표/성공 기준을 첫 줄에 명시하고, 이후 제안 작성"

### 자주하는 실수
환경·버전·정책·예산·시간 정보 누락, 성공 기준 부재

### 더 좋게
금지사항(Do not)·우선순위(Must/Should/Could)를 구분 표기

### 성공 기준 예시
정책 위반 0건, 시간 박스 준수, 오류율 <1% 등 측정 가능한 기준 포함

참고: 제약 조건의 상세 내용은 [5장. 역할(Agent)과 제약(Constraint) 설계](05-agent-constraints.md)에서 자세히 다룹니다.

## 3.5 실패하는 지시의 특징
실패 패턴을 알면 즉시 교정할 수 있습니다.

### 언제 유용한가
지시 품질 셀프 리뷰·팀 코드리뷰/문서리뷰 체크리스트로 활용할 때

### 언제 안 쓰나
탐색/아이디어 발산 자체가 목표일 때(일시 허용)

### 예시
나쁜 지시 → “이거 좀 좋게” / 좋은 지시 → “대상: … 형식: … 길이: … 성공 기준: …”

- (개발) 좋은 지시: "대상: 모듈 X, 범위: 함수 3개, 형식: PR + 변경요약 5줄, 길이: ~200줄, 성공기준: 테스트 100%·복잡도 20%↓"
- (마케팅) 좋은 지시: "대상: 20–30대, 톤: 경쾌, 형식: 헤드라인 3개(각 20자 내), 성공기준: USP 반영·중복어구 없음"
- (운영) 좋은 지시: "형식: 요약 3줄·표 1개, 기간: 전주, 성공기준: 지표 3개·이슈 1개·액션 1개 포함"

### 나쁜예
- 모호어 남발(적절히/빠르게/좋게)
- 범위 무제한(대상/기간/자료원 미정)
- 형식·길이 없음
- 성공 기준 없음

- (개발) "코드 깔끔하게"
- (마케팅) "좋은 카피 써줘"
- (운영) "리포트 깔끔히"

### 이렇게 적어보세요
"대상·범위·형식·길이·성공 기준 5요소를 1줄에 요약하고, 이어서 상세 지시를 적기"

### 자주하는 실수
