# 용어 전환 가이드

**목적**: 2권에서 사용하는 새로운 용어를 이해하고, 기존 용어와의 차이를 명확히 합니다.

---

## 용어 변경 철학

2권은 **사고 중심(Thinking-First)** 접근을 강조합니다. AI 시대의 조직은 실행이 아닌 **사고 프로세스**를 설계해야 합니다. 이러한 철학을 반영하기 위해 용어를 변경했습니다.

---

## 핵심 용어 매핑

| 기존 용어 | 새 용어 | 변경 이유 |
|----------|---------|----------|
| 단위 조직 (Unit Organization) | **사고 클러스터** (Thinking Cluster) | 조직 구조보다 사고 프로세스 강조 |
| 복합 조직 (Complex Organization) | **계층적 사고 클러스터** (Hierarchical Thinking Cluster) | 일관된 명명 체계, 사고 중심 |
| 아키텍트 (Architect) | **사고 조율자** (Thinking Coordinator) | 설계보다 조율의 역할 명확화 |
| 워커 (Worker) | **실행 워커** (Execution Worker) | 실행 담당 명시 |
| 메타 에이전트 (Meta Agent) | **메타 조율자** (Meta Coordinator) | 일관된 조율자 명명 |

---

## 용어별 상세 설명

### 1. 사고 클러스터 (Thinking Cluster)

**정의**: 복잡한 목표를 관리 가능한 사고 단위로 분해하고, 인간과 AI가 함께 사고하고 추론하는 단위

**왜 "조직"이 아닌 "클러스터"인가?**

- **조직(Organization)**: 구조와 계층, 역할 분담을 의미
- **클러스터(Cluster)**: 관련된 것들의 집합, 유연하고 동적

AI 시대의 핵심은 **누가 무엇을 하는가**(조직 구조)가 아니라 **어떻게 생각하고 판단하는가**(사고 프로세스)입니다. "사고 클러스터"는 이러한 사고 프로세스의 집합을 의미합니다.

**사용 예시:**

```markdown
❌ 나쁜 예 (조직 중심):
"이 단위 조직이 콘텐츠를 생성합니다"

✅ 좋은 예 (사고 중심):
"이 사고 클러스터가 콘텐츠 생성을 위한 사고 프로세스를 설계합니다"
```

```markdown
❌ 나쁜 예:
"워커들을 조직에 배치합니다"

✅ 좋은 예:
"사고 클러스터 내에서 사고 단계별로 역할을 정의합니다"
```

---

### 2. 계층적 사고 클러스터 (Hierarchical Thinking Cluster)

**정의**: 여러 사고 클러스터를 조율하여 더 복잡한 목표를 달성하는 구조

**왜 "복합 조직"이 아닌 "계층적 사고 클러스터"인가?**

- 일관된 명명 체계 유지
- 여전히 사고 프로세스에 집중
- 계층(Hierarchical)은 구조적 관계를 명확히 표현

**사용 예시:**

```markdown
❌ 나쁜 예:
"복합 조직이 여러 단위 조직을 관리합니다"

✅ 좋은 예:
"계층적 사고 클러스터가 여러 사고 클러스터의 사고 흐름을 조율합니다"
```

---

### 3. 사고 조율자 (Thinking Coordinator)

**정의**: 사고 클러스터를 설계하고 조율하는 역할, 주로 인간이 담당

**왜 "아키텍트"가 아닌 "조율자"인가?**

- **아키텍트(Architect)**: 건축, 설계의 이미지 → 정적
- **조율자(Coordinator)**: 조율, 조정의 이미지 → 동적

AI 시대의 역할은 한 번 설계하고 끝나는 것이 아니라, 지속적으로 **조율하고 조정**하는 것입니다. 인간과 AI의 협업을 조율한다는 의미가 더 강합니다.

**사고 조율자의 역할:**

- 전체 사고 흐름 설계
- 의사결정 지점 관리
- 품질 기준 정의
- 예외 상황 처리

**사용 예시:**

```markdown
❌ 나쁜 예:
"아키텍트가 작업을 워커에게 할당합니다"

✅ 좋은 예:
"사고 조율자가 사고 프로세스를 설계하고 실행 워커와 협업합니다"
```

---

### 4. 실행 워커 (Execution Worker)

**정의**: 정의된 사고 프로세스에 따라 분석과 실행을 담당하는 역할, 주로 AI가 담당

**왜 "워커"에서 "실행 워커"로?**

- 역할을 더 명확히: **실행**을 담당
- 사고 조율자와의 대비: 조율 vs 실행

**실행 워커의 역할:**

- 데이터 분석 및 패턴 인식
- 규칙 기반 추론
- 콘텐츠 생성 및 변환
- 반복 작업 수행

**사용 예시:**

```markdown
❌ 나쁜 예:
"워커가 콘텐츠를 생성합니다"

✅ 좋은 예:
"실행 워커가 사고 조율자가 설계한 프로세스에 따라 콘텐츠를 생성합니다"
```

---

### 5. 메타 조율자 (Meta Coordinator)

**정의**: 여러 사고 클러스터를 조율하는 상위 역할

**왜 "메타 에이전트"가 아닌 "메타 조율자"인가?**

- 일관된 "조율자" 명명 체계
- 역할의 본질(조율) 강조

**메타 조율자의 역할:**

- 여러 사고 클러스터 간 조율
- 전체 목표와의 정렬
- 리소스 분배
- 충돌 해결

**사용 예시:**

```markdown
❌ 나쁜 예:
"메타 에이전트가 하위 에이전트를 관리합니다"

✅ 좋은 예:
"메타 조율자가 여러 사고 클러스터의 사고 흐름을 조율하고 전체 목표 달성을 돕습니다"
```

## 용어 사용 가이드라인

### DO: 이렇게 사용하세요

✅ 사고 프로세스를 강조할 때:
- "사고 클러스터가 문제 해결 프로세스를 설계합니다"
- "사고 조율자가 사고 흐름을 조율합니다"

✅ 인간과 AI의 역할을 구분할 때:
- "사고 조율자(인간)는 전략을, 실행 워커(AI)는 실행을 담당합니다"

✅ 계층 관계를 설명할 때:
- "메타 조율자가 여러 사고 클러스터를 조율합니다"

### DON'T: 이렇게 사용하지 마세요

❌ 조직 구조를 강조:
- "사고 클러스터가 팀원을 관리합니다" (X)
- → "사고 클러스터가 사고 단계를 정의합니다" (O)

❌ 구 용어와 신 용어 혼용:
- "아키텍트가 사고 클러스터를 설계합니다" (X)
- → "사고 조율자가 사고 클러스터를 설계합니다" (O)

## 빠른 참조표

### 인간이 하는 것 → "조율"
- 사고 조율자
- 메타 조율자

### AI가 하는 것 → "실행"
- 실행 워커

### 사고 단위 → "클러스터"
- 사고 클러스터
- 계층적 사고 클러스터
