# 1장. 프롬프트와 인스트럭션 이해하기

**Part 1: 프롬프트와 인스트럭션의 기초**

### 이 장에서 배우는 것
- 프롬프트와 인스트럭션의 기본 개념
- 좋은 프롬프트를 작성하는 법과 그 한계
- 프롬프트를 잘 쓰기 위한 핵심 요소(명확성·구체성·맥락·형식·제약)
- 프롬프트 사용 방식의 차이와 인스트럭션의 필요성

## 1.1 프롬프트란?

[프롬프트(Prompt)](glossary.md#프롬프트-prompt)는 AI 모델에게 말을 거는 방식이자, 우리가 원하는 것을 얻어내기 위한 '지시문'입니다. 이는 마치 우리가 새로운 외국어를 배우듯, AI라는 새로운 지적 파트너와 소통하는 방법을 배우는 것과 같습니다. 단순한 질문부터 복잡한 요청, 상세한 설명과 예시까지 다양한 형태를 가질 수 있습니다.

## 1.2 프롬프트를 잘 사용하려면

효과적인 프롬프트는 AI가 사용자의 의도를 정확히 파악하고 원하는 결과물을 만들어낼 수 있도록 돕는 '좋은 질문'입니다. 좋은 프롬프트를 작성하기 위해서는 다섯 가지 핵심 요소를 갖추어야 합니다.

### 1) 명확성과 구체성

가장 기본이 되는 것은 **명확성**과 **구체성**입니다. 단순히 "기후 변화에 대해 알려줘"라고 묻기보다 "최근 10년간 해수면 상승에 영향을 미친 3가지 요점을 설명해줘"처럼 구체적으로 질문하면 훨씬 더 유용한 답변을 얻을 수 있습니다.

### 2) 맥락과 목적

여기에 **맥락**과 **목적**을 더하면 결과물의 품질을 더 높일 수 있습니다. AI가 사용자의 상황을 이해하고 작업의 최종 목표를 알 때, 더 적절한 결과물을 생성하기 때문입니다. "이 코드를 개선해줘"라는 요청보다는 "저는 파이썬 초보 개발자입니다. 아래 코드는 분석 속도가 너무 느린데, 실행 속도를 개선하고 그 이유를 1~2줄로 설명해주세요."와 같이 배경 정보와 목적을 함께 제공하는 것이 좋습니다.

### 3) 예시와 역할

원하는 결과물의 **예시**를 한두 개 제공하는 것도 AI가 사용자의 의도를 파악하는 데 큰 도움이 됩니다. 이는 AI가 소수의 예시만으로도 패턴을 학습하여 새로운 문제를 해결하는 'Few-shot learning'[^1] 원리를 활용하는 것입니다. 또한, AI에게 특정 전문가의 **역할**을 부여하면(예: "당신은 데이터 과학자입니다."), 답변의 톤과 관점을 원하는 방향으로 유도할 수 있습니다.

### 4) 형식과 제약 조건

마지막으로, 결과물의 **형식**을 명확히 지정하고(예: "마크다운 표 형식으로"), 지켜야 할 **제약 조건**을 명시하면(예: "300단어 이내로"), AI는 사용자가 원하는 구조에 맞춰 예측 가능한 결과물을 생성하게 됩니다.

### 5) 피해야 할 표현

반대로, 좋은 프롬프트 작성을 방해하는 표현들도 있습니다. "충분히", "적절히"와 같이 기준이 모호한 형용사를 피하고, 구체적인 수량이나 기준을 명시하는 것이 좋습니다. 또한, 작업의 목적과 범위를 명확히 하고, 원하는 결과물의 형식과 성공 기준을 제시해야 AI가 방황하지 않고 정확한 결과물을 만들어낼 수 있습니다.

이러한 요소들은 이후 장에서 템플릿과 함께 더 자세히 다룰 것입니다.

💡 **여기까지의 정리**

자, 여기까지 프롬프트를 효과적으로 작성하는 5가지 핵심 요소를 배웠습니다. 이 원칙들을 잘 활용하면 AI로부터 원하는 결과를 얻을 수 있죠.

그런데 실제로 이런 프롬프트를 매일 사용하다 보면 한 가지 의문이 생깁니다.

## 1.3 프롬프트 사용 방식의 차이

프롬프트 자체는 훌륭한 도구입니다. 하지만 실제 업무에서는 **어떻게 사용하느냐**가 더 중요합니다. 같은 내용의 프롬프트라도 사용 방식에 따라 전혀 다른 결과를 만들어냅니다.

### 두 가지 사용 방식

업무 현장에서 프롬프트는 크게 두 가지 방식으로 사용됩니다.

**방식 1: 즉흥적 사용**

필요할 때마다 채팅창에 타이핑해서 사용하는 방식입니다.

```
[월요일]
"주간 판매 보고서 만들어줘"
→ AI가 보고서 생성
→ 좋네! 사용 완료

[다음 주 월요일]
"주간 판매 보고서... 어떻게 했더라?"
→ 채팅 기록 뒤지기
→ 비슷하게 다시 작성
→ 조금 다른 결과물
```

**방식 2: 체계적 사용**

프롬프트를 파일로 저장하고, 필요할 때 실행하는 방식입니다.

```
[처음 한 번]
📄 weekly-sales-report.instruction.md 파일 작성
- 어떤 데이터를 받을지
- 어떻게 처리할지
- 어떤 형식으로 만들지
→ 파일로 저장

[매주 월요일]
파일 실행
→ 항상 같은 형식의 보고서
→ 팀원들도 같은 파일 사용 가능
```

### 두 방식의 차이

|  | 즉흥적 사용 | 체계적 사용 |
|---|-----------|-----------|
| **저장** | 채팅 기록에 묻힘 | 파일로 영구 보관 |
| **재사용** | 매번 다시 작성 또는 복붙 | 파일만 실행 |
| **일관성** | 실행할 때마다 조금씩 다름 | 항상 같은 형식 |
| **공유** | 메신저로 복붙 전달 | 파일 공유 |
| **개선** | 이전 버전 찾기 어려움 | Git으로 변경 이력 관리 |
| **협업** | 각자 조금씩 다르게 작성 | 팀 전체가 같은 기준 사용 |

차이의 핵심은 단순합니다. **프롬프트를 어디에 쓰느냐**입니다.

- 채팅창에 쓰면 → 휘발되고, 찾기 어렵고, 매번 새로 만들어야 합니다
- 파일에 쓰면 → 보관되고, 재사용되고, 팀과 공유할 수 있습니다

### 왜 이 차이가 중요한가?

간단한 작업이라면 채팅창에서 즉흥적으로 사용해도 충분합니다. 하지만 다음과 같은 상황이라면 체계적 사용이 필요합니다:

**반복되는 작업**
```
매주 같은 보고서를 만든다면?
→ 매번 새로 작성하는 건 비효율적
→ 파일로 만들어서 재사용
```

**팀과 함께하는 작업**
```
팀원 10명이 같은 방식으로 작업해야 한다면?
→ 구두 설명으로는 각자 다르게 이해
→ 파일로 공유하면 모두 같은 기준
```

**계속 개선해야 하는 작업**
```
이번 주 보고서를 다음 주에 더 나아지게 만들려면?
→ 뭘 바꿨는지 기억하기 어려움
→ 파일로 관리하면 이력 추적 가능
```

### 인스트럭션: 프롬프트의 체계적 사용

프롬프트를 체계적으로 사용하는 방식, 즉 파일로 만들어서 재사용하고 팀과 공유하고 지속적으로 개선하는 방식을 우리는 **[인스트럭션(Instruction)](glossary.md#인스트럭션-instruction)**이라고 부릅니다.

인스트럭션은 단순히 "프롬프트를 파일로 저장한 것"이 아닙니다. 체계적으로 사용하기 위해서는 특별한 구조가 필요합니다. 이 구조를 갖추었을 때, 프롬프트는 개인의 도구를 넘어 **시스템**이 됩니다.

---

💡 **비유: 요리와 레시피**

**즉흥적 사용 = 기억에 의존한 요리**
- 오늘은 맛있게 됐는데 내일은 좀 다르고
- "소금 적당히, 설탕 조금" → 매번 달라짐
- 다른 사람한테 설명하기 어려움

**체계적 사용 = 레시피 카드로 요리**
- "소금 2g, 설탕 5g, 180도 20분" → 정확히 기록
- 언제 누가 만들어도 같은 맛
- 레시피 카드만 있으면 누구나 재현 가능

프롬프트도 마찬가지입니다. 채팅창에서 즉흥적으로 사용하는 것과, 파일로 체계화해서 사용하는 것은 완전히 다른 결과를 만들어냅니다.

---

### 시스템이란 무엇인가?

인스트럭션이 단순한 파일이 아니라 **시스템**이라고 했습니다. 그렇다면 시스템이란 무엇일까요?

시스템은 다음과 같은 특징을 가진 구조입니다:

**1) 재사용 가능**: 한 번 만들면 계속 쓸 수 있다  
**2) 일관성 보장**: 같은 입력이면 같은 결과가 나온다  
**3) 개선 가능**: 시간이 지나면서 더 나아진다  
**4) 협업 가능**: 여러 사람이 함께 사용할 수 있다  
**5) 확장 가능**: 규모가 커져도 작동한다

이 다섯 가지가 바로 좋은 시스템의 핵심 속성입니다. 인스트럭션도 이 속성들을 만족할 때 진짜 시스템이 됩니다.

다음 장에서는 질문을 설계하는 다양한 방법을 배우고, [3장. 효과적인 지시 작성의 기본 원칙](vol-1-part-1-chapter-03.md)에서는 이러한 시스템 속성을 만족하는 인스트럭션을 어떻게 만드는지 구체적으로 살펴보겠습니다.

**프롬프트의 5가지 핵심 요소**로 명확성, 구체성, 맥락, 형식, 제약 조건을 배웠고, 이를 통해 AI로부터 원하는 결과를 얻는 방법을 이해했습니다. 

**핵심 통찰**:
프롬프트 자체는 훌륭한 도구입니다. 하지만 **어떻게 사용하느냐**가 더 중요합니다.

**두 가지 사용 방식**:
- **즉흥적 사용**: 채팅창에서 매번 새로 작성 (간단한 작업에 적합)
- **체계적 사용**: 파일로 저장해서 재사용 (반복 작업, 팀 협업에 필수)

**인스트럭션**:
프롬프트를 체계적으로 사용하는 방식입니다. 단순한 파일이 아니라, 재사용 가능하고, 일관성 있고, 개선 가능하고, 협업 가능하고, 확장 가능한 **시스템**입니다.

다음 장에서는 이러한 인스트럭션을 실제로 만드는 방법을 구체적으로 배우게 됩니다.

## 참고: 주요 프롬프팅 기법들

AI와의 상호작용은 단순한 질문-답변을 넘어 다양한 기법으로 발전해왔습니다. 대표적인 프롬프팅 기법들은 다음과 같습니다:

### Zero-shot 프롬프팅
예시 없이 바로 작업을 요청하는 방식입니다.
- 예: "번역해줘: Hello world"
- 특징: 가장 간단하지만 복잡한 작업에는 한계가 있음

### Few-shot 프롬프팅
AI에게 몇 가지 예시를 먼저 제공하여 패턴을 학습시키는 방식입니다.
- 예: "예시: 사과→apple, 바나나→banana. 그럼 오렌지는?"
- 특징: 패턴이 명확한 작업에 효과적

### Chain-of-Thought (CoT) 프롬프팅
AI가 단계별로 사고 과정을 보여주도록 유도하는 방식입니다.
- 예: "이 문제를 단계별로 생각해보자. 먼저..."
- 특징: 복잡한 추론이나 수학 문제 해결에 유용

### Role 프롬프팅
AI에게 특정 전문가나 역할을 부여하는 방식입니다.
- 예: "당신은 10년 경력의 데이터 과학자입니다."
- 특징: 답변의 톤, 관점, 전문성 수준을 조정할 수 있음

이러한 기법들은 상황에 따라 선택적으로 활용되며, 이후 인스트럭션 설계에서도 핵심 구성 요소가 됩니다.

### 실습 과제

**과제 1: 5가지 핵심 요소로 프롬프트 개선하기**

다음 단순한 프롬프트를 5가지 핵심 요소를 모두 포함하도록 개선하세요:

*원본:* "회의록을 요약해줘"

*개선 방향:*
1. 명확성과 구체성: 어떤 회의록인지, 요약 길이와 포함할 항목
2. 맥락과 목적: 요약의 목적, 누가 사용할지
3. 예시와 역할: (필요시) 원하는 요약 스타일 예시
4. 형식과 제약 조건: 결과물 구조, 반드시 지켜야 할 조건

**과제 2: 프롬프팅 기법 비교**
