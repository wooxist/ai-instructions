## 프롬프트에서 시스템으로

1장과 2장을 통해 우리는 좋은 프롬프트의 원칙을 배우고, 이를 효과적인 질문으로 구체화하는 방법을 익혔습니다. 이제 개별적으로는 AI와 잘 소통할 수 있게 되었죠.

하지만 실무에서는 새로운 도전이 기다리고 있습니다:

**김대리의 하루**
```
09:00 - "주간 보고서 만들어야지" → 지난주 프롬프트 찾기 (10분)
10:00 - "고객 분석 해야지" → 어떻게 했더라? 다시 작성 (15분)  
14:00 - "신입이 물어보네" → 구두로 설명 (30분)
15:00 - "팀장님이 형식 통일하래" → 각자 다른 방식... (혼란)
```

개인의 스킬은 있지만 시스템이 없을 때 발생하는 문제들입니다.

**진짜 해결책은 무엇일까요?**

답은 '시스템화'에 있습니다. 개별 프롬프트를 넘어서 조직의 자산이 되는 체계를 만드는 것이죠.

이 장에서는 프롬프트를 진정한 '시스템'으로 만드는 방법을 배웁니다. 먼저 시스템이 갖춰야 할 5가지 핵심 속성을 이해하고, 이를 실현하는 9가지 구성 요소를 단계적으로 살펴보겠습니다.

## 3.1 인스트럭션: 시스템의 5가지 핵심 속성

시스템이 되려면 무엇이 필요할까요? 건물을 짓는 것처럼, 시스템도 단계적으로 구축됩니다. 

### 시스템 구축의 3단계 레이어

**왜 레이어로 나누는가?**
마치 매슬로의 욕구 단계처럼, 시스템도 기초가 튼튼해야 다음 단계로 나아갈 수 있습니다. 기초 없이 상위 단계를 추구하면 모래성과 같습니다.

좋은 시스템은 다섯 가지 핵심 속성을 가지고 있습니다. 이 속성들은 독립적이지 않고, 세 가지 레이어로 계층을 이룹니다:

**🏗️ 기반 레이어**: 시스템의 최소 조건
1. **재사용 가능** - 반복 사용의 기반
2. **일관성 보장** - 신뢰할 수 있는 결과

이 두 가지가 없으면 시스템이라고 부를 수 없습니다. 한 번만 쓰고 버리거나, 매번 다른 결과가 나온다면 의미가 없으니까요.

**📈 성장 레이어**: 시스템의 진화
3. **개선 가능** - 시간에 따라 발전
4. **협업 가능** - 공간에 따라 확산

기반이 탄탄하면 이제 시스템을 더 나아지게 만들 수 있습니다. 혼자서 점점 개선하고, 팀 전체로 확산되는 거죠.

**🚀 확장 레이어**: 시스템의 스케일
5. **확장 가능** - 규모와 범위로 성장

성장이 안정화되면 이제 더 큰 규모, 더 넓은 범위로 확장할 수 있습니다.

이 계층 구조를 염두에 두고 각 속성을 하나씩 살펴보겠습니다.

### 1) 재사용 가능 (Reusable)

첫 번째 속성은 재사용 가능성입니다. 한 번 만들면 계속해서 반복적으로 사용할 수 있어야 한다는 뜻이죠.

채팅창에서 프롬프트를 쓸 때를 떠올려보세요. 이번 주에 "주간 판매 보고서 만들어줘"라고 작성했다면, 다음 주에도 비슷한 보고서가 필요할 때 어떻게 하시나요? 대부분은 지난 채팅 기록을 뒤적이거나, 기억을 더듬어 다시 작성합니다.

```
[이번 주]
"주간 판매 보고서 만들어줘. 표로 해주고, 증감률도 넣어줘..."

[다음 주]
"주간 판매 보고서... 어떻게 했더라? 아, 표로 하고, 증감률도 넣고... 
근데 뭘 더 넣었더라?"
→ 매번 비슷한 작업을 하면서도 처음부터 다시 작성
```

매번 비슷한 작업을 하면서도 처음부터 다시 작성하는 이유는 간단합니다. 프롬프트가 휘발되기 때문이죠. 하지만 파일로 만들어두면 상황이 완전히 달라집니다.

```markdown
📄 weekly-sales-report.instruction.md

## 입력
파일: sales_{YYYYMMDD}.csv

→ 입력 데이터만 바꾸면 계속 사용할 수 있습니다
```

입력 데이터만 바꾸면 계속 사용할 수 있습니다. 재사용 가능성을 높이는 핵심은 특정 상황에 의존하는 값들을 하드코딩하지 않고 파라미터로 만드는 것입니다. "2024년 3월 서울 지점"처럼 구체적인 값을 박아넣지 말고, "{연월} {지점명}"처럼 변수로 만들어두는 거죠.

### 2) 일관성 보장 (Consistent/Idempotent)

두 번째 속성은 일관성입니다. 같은 입력이면 언제, 누가 실행해도 같은 형식의 결과가 나와야 합니다.

이런 경험 있으신가요? 월요일 오전에 "보고서 만들어줘"라고 했을 때는 5페이지짜리 상세 분석이 나왔는데, 금요일 오후에 똑같이 요청했더니 2페이지짜리 간단한 요약만 나왔습니다. "왜 다르지?"라고 의아해하면서도 이유를 찾기 어렵죠.

```
[월요일 오전]
"보고서 만들어줘"
→ 5페이지 상세 분석

[금요일 오후]
"보고서 만들어줘"
→ 2페이지 간단 요약

왜 다르지?
```

이런 비일관성이 발생하는 이유는 "보고서"가 구체적으로 어떤 형태여야 하는지 명시되지 않았기 때문입니다. AI는 매번 조금씩 다르게 해석하고, 그 결과도 달라집니다.

이 문제를 해결하려면 출력 형식을 명확히 정의해야 합니다.

```markdown
## 출력
- 형식: 마크다운
- 길이: 500단어 (±50단어)
- 구조: 3개 섹션 (요약/분석/제안)

→ 이제 언제 실행해도 같은 형식
```

"적절히", "충분히", "대략" 같은 모호한 표현을 모두 제거하고, 구체적인 수치와 기준을 명시하는 것이 핵심입니다.

### 3) 개선 가능 (Improvable)

세 번째 속성은 개선 가능성입니다. 시간이 지나면서 점점 더 나아질 수 있어야 한다는 뜻이죠.

업무를 하다 보면 프롬프트를 조금씩 개선하게 됩니다. "아, 이번엔 증감률도 넣어야겠다", "길이가 너무 길어서 500자로 제한해야겠다" 같은 식으로요. 그런데 채팅창에서는 이런 개선 과정이 기록되지 않습니다.

```
1주차: "보고서 만들어줘"
2주차: "보고서 만들어줘. 아, 증감률도 넣어줘"
3주차: "보고서... 증감률... 뭘 더 넣었더라?"
```

무엇을 어떻게 바꿨는지, 왜 바꿨는지 추적하기가 어렵습니다. 3주 후에는 2주차에 뭘 추가했는지조차 기억나지 않죠.

파일로 관리하면 Git 같은 버전 관리 시스템으로 변경 이력을 추적할 수 있습니다.

```bash
git log weekly-sales-report.instruction.md

v1.0 (2024-01): 초기 버전
v2.0 (2024-02): 증감률 컬럼 추가 (CFO 요청)
v2.1 (2024-03): 500자 제한 추가 (보고서 표준화)
```

각 변경마다 왜 바꿨는지 이유를 기록해두면, 나중에 "이게 왜 이렇게 되어 있지?"라는 고민을 할 필요가 없습니다. 개선의 맥락을 이해할 수 있고, 필요하다면 이전 버전으로 되돌리는 것도 가능하죠.

### 4) 협업 가능 (Collaborative)

네 번째 속성은 협업 가능성입니다. 여러 사람이 함께 사용하고, 함께 개선할 수 있어야 합니다.

채팅창의 프롬프트는 본질적으로 개인의 것입니다. 당신이 만든 훌륭한 프롬프트를 동료와 공유하려면 어떻게 해야 할까요?

```
동료: "그 보고서 어떻게 만드셨어요?"
나: "음... 이렇게 저렇게... (구두로 설명)"
동료: "아 네..." (실제로는 70%만 이해)

→ 결국 각자 다른 방식으로 작업
```

구두로 설명하거나 메신저로 텍스트를 복사해서 보내줄 수는 있지만, 이건 진정한 협업이 아닙니다. 동료가 당신의 설명을 100% 이해했는지 확인할 방법도 없고, 동료가 프롬프트를 개선했을 때 그 변경사항을 다시 당신에게 전달하기도 어렵습니다.

파일 기반 인스트럭션은 다릅니다.

```markdown
📄 weekly-sales-report.instruction.md
→ Git 저장소에 커밋
→ 팀 전체가 같은 파일 사용
→ 신입도 파일만 보고 실행 가능
→ Pull Request로 함께 개선
```

파일 하나를 공유하면 팀 전체가 같은 기준으로 작업할 수 있습니다. 누군가 개선 아이디어가 있으면 Pull Request를 만들고, 팀에서 리뷰하고, 승인되면 모두가 개선된 버전을 사용합니다. 이것이 진정한 협업이죠.

협업 가능성을 높이는 핵심은 암묵적 지식에 의존하지 않는 것입니다. "이건 당연히 알겠지"라고 가정하지 말고, 모든 것을 명시적으로 문서화해야 합니다.

### 5) 확장 가능 (Scalable)

다섯 번째이자 마지막 속성은 확장 가능성입니다. 데이터 규모가 커지거나 적용 범위가 넓어져도 계속 작동해야 합니다.

채팅창에서 즉흥적으로 작성하다 보면 현재 상황에 딱 맞춘 프롬프트가 되기 쉽습니다.

```
이번 주: "서울 지역 3월 데이터 분석해줘"
다음 주: "부산 지역도 분석해줘" → 새로 작성
다다음 주: "전국 분석해줘" → 또 새로 작성

문제: 매번 새로 만들어야 함
```

서울에서 부산으로, 부산에서 전국으로 확장될 때마다 프롬프트를 새로 작성해야 합니다. 비효율적이죠.

파일로 만들면 처음부터 확장 가능한 구조로 설계할 수 있습니다.

```markdown
📄 sales-analysis.instruction.md

## 입력
- region: 지역 코드 (ALL, SEL, BUS, DGU, ...)
- month: 분석 월 (YYYYMM)

→ 어떤 지역이든, 어떤 월이든 적용 가능
→ 새로운 지역이 추가되어도 파일 수정 불필요
```

확장 가능성을 높이는 핵심은 특정 상황이나 데이터에 종속되지 않도록 일반화된 로직으로 만드는 것입니다. "서울"이라고 박아넣지 말고 "지역"이라는 파라미터로 만들고, "3월"이 아니라 "월"로 만드는 식이죠.

---

💡 **5가지 속성을 다시 정리하면**

좋은 시스템의 핵심 속성 다섯 가지를 살펴봤습니다:

1. **재사용 가능**: 한 번 만들면 계속 쓴다
2. **일관성 보장**: 같은 입력이면 같은 출력이 나온다
3. **개선 가능**: 시간이 지나면서 더 나아진다
4. **협업 가능**: 여러 사람이 함께 쓰고 개선한다
5. **확장 가능**: 규모가 커지고 범위가 넓어져도 작동한다

인스트럭션은 바로 이 5가지 속성을 모두 만족하는 프롬프트 시스템입니다.

여기서 자연스럽게 다음 질문이 떠오릅니다. **"이 5가지 속성을 실제로 어떻게 만족시킬 수 있을까?"**

답은 명확한 구조에 있습니다.

## 3.2 시스템 속성을 만족하는 9가지 구성 요소

5가지 시스템 속성을 만족하려면 프롬프트에 명확한 구조가 필요합니다. 그냥 파일로 저장한다고 해서 자동으로 시스템이 되는 건 아니니까요.

이 구조를 이해하려면 먼저 채팅창과 파일의 근본적인 차이를 이해해야 합니다.

### 채팅창에서는 대화로 빈칸을 채운다

채팅창에서 프롬프트를 사용할 때는 이렇게 진행됩니다:

```
나: "보고서 만들어줘"
AI: "어떤 형식으로 할까요?"
나: "표로 해줘"
AI: "몇 개 컬럼이 필요하신가요?"
나: "3개로 해줘"

→ 대화하면서 점점 구체화
→ 빠진 부분을 실시간으로 채워나감
```

이건 실제로 꽤 편리합니다. 처음부터 완벽하게 작성할 필요 없이, AI가 물어보면 그때그때 답하면 되니까요. 하지만 이 방식에는 치명적인 문제가 있습니다. 바로 **재사용할 수 없다**는 점이죠.

### 파일로 만들면 재사용과 공유가 가능하다

채팅창의 문제는 휘발성입니다. 매번 대화로 빈칸을 채워야 하죠. 하지만 파일로 저장하면 상황이 달라집니다. 저장된 인스트럭션은 반복 사용할 수 있고, 팀원들과 공유할 수 있습니다.

그런데 여기서 고려해야 할 것이 있습니다.

### 내용이 불충분하면 재사용/공유 가치가 사라진다

파일로 만들었다고 끝이 아닙니다. 내용이 불충분하면 어떻게 될까요?

```markdown
📄 report.instruction.md (v1.0)
"보고서 만들어줘"
```

이렇게 불충분한 내용의 인스트럭션을 파일로 만들고 실행하는 상황을 생각해보죠:

```
→ 파일 실행
→ AI: "어떤 형식으로 할까요?"
→ 나: "표로 해줘"
→ AI: "몇 개 컬럼이 필요하신가요?"
→ 나: "3개로 해줘"
```

작동은 합니다. 하지만 여기서 문제가 발생합니다.

**재사용성 문제**:
```
이번 주: 파일 실행 + "표로 해줘" 대화
다음 주: 파일만 실행 → AI: "어떤 형식으로 할까요?"
       → "아, 또 물어보네?"
       → 매번 같은 대화 반복
```

**일관성 문제**:
```
A 팀원: 파일 실행 + "표로 3개 컬럼"
B 팀원: 파일 실행 + "불릿 리스트로 5개 항목"
→ 같은 파일인데 완전히 다른 결과
```

**협업 문제**:
```
파일: "보고서 만들어줘"
신입: "이거 어떻게 쓰는 거예요?"
선임: "아, 실행하면 AI가 물어보는데, 그때 이렇게 저렇게..."
→ 파일만으로는 불충분하고 구두 설명이 필요
```

보이시나요? 파일로 만들었지만 내용이 불충분하면 채팅창과 다를 바가 없습니다. 매번 추가 대화가 필요하고, 사람마다 다르게 답변하면서 결과도 달라집니다.

### 충분히 명확한 인스트럭션이란?

따라서 인스트럭션은 **추가 대화 없이도 실행 가능한 수준**으로 명확해야 합니다. 그래야 진짜 재사용하고 공유할 수 있습니다. AI가 추가로 물어볼 필요가 없을 정도로 모든 것이 명시되어 있어야 하죠.

물론 처음부터 완벽할 필요는 없습니다. 사실 처음부터 완벽하게 만드는 것도 불가능합니다. 실제로 사용해봐야 뭐가 부족한지 알 수 있으니까요.

실용적인 접근은 이렇습니다:

```
v1.0: 최소한만 작성 (필수 5가지)
      → 사용해보니 결과가 매번 다르네?
v1.1: 출력 형식을 더 구체적으로 명시
      → 사용해보니 금지사항을 어기는 경우가 있네?
v1.2: 제약 조건 추가
      → 사용해보니 품질이 들쑥날쑥하네?
v2.0: 평가 기준 추가
```

이렇게 **점진적으로 개선**하면서, 결국 "충분히 명확한" 상태에 도달하는 것이 목표입니다.

### 명확성의 4단계 스펙트럼

"충분히 명확하다"는 것은 상대적입니다. 여러분의 인스트럭션이 현재 어느 수준인지 확인해보세요:

#### Level 0: 채팅 의존 (시스템 아님)
```
"보고서 만들어줘"
```
→ AI 반응: "어떤 형식으로 하시겠습니까?" (3회 이상 질문)  
→ 문제: 매번 대화로 빈칸 채우기, 재사용 불가능

#### Level 1: 최소 실행 가능 (시스템의 시작)
```markdown
목적, 역할, 입력, 처리, 출력 명시 (필수 5가지)
```
→ AI 반응: "확인했습니다" (바로 실행 가능)  
→ 특징: 실행은 되지만 결과가 조금씩 다를 수 있음  
→ 적용: v1.0 단계

#### Level 2: 안정적 품질 (실전 사용)
```markdown
Level 1 + 제약 조건 + 평가 기준 (필수 5 + 권장 2)
```
→ AI 반응: 조용히 실행하고 체크리스트로 자가 검증  
→ 특징: 일관된 품질, 실패율 10% 미만  
→ 적용: v2.0 단계, 팀 공유 시작

#### Level 3: 완전 명확 (성숙한 시스템)
```markdown
Level 2 + 기초 지식 + 출력 템플릿 (9가지 전체)
```
→ AI 반응: 완벽히 이해하고 예시와 동일한 품질로 출력  
→ 특징: 신입도 설명 없이 사용 가능, 실패율 1% 미만  
→ 적용: v3.0+, 조직 표준화

**판단 기준**:
- Level 1 미만: 아직 시스템이 아님 → 필수 5가지부터 채우기
- Level 1-2: 개인 사용은 OK → 팀 공유 전 Level 2로 올리기
- Level 2-3: 팀 사용 안정 → 필요시 Level 3으로 성숙화

대부분의 경우 **Level 2면 충분**합니다. Level 3은 조직 전체 표준이나 장기 운영 시스템에만 필요합니다.

그렇다면 처음 시작할 때 "최소한"은 무엇일까요? 그리고 "충분히 명확한" 상태가 되려면 어떤 요소들이 필요할까요?

바로 이것이 9가지 구성 요소입니다.

### 9가지 구성 요소의 역할

각 구성 요소는 시스템의 특정 속성을 만족시킵니다:

| 구성 요소 | 만족시키는 속성 | 왜 필요한가 |
|---------|---------------|-----------|
| **1. 목적** | 협업, 개선 | 팀 전체가 같은 목표를 공유하고, 개선 방향을 설정 |
| **2. 역할** | 일관성 | 답변의 톤과 깊이를 일관되게 유지 |
| **3. 입력** | 재사용, 확장 | 파라미터화로 반복 사용 가능, 다른 데이터에도 적용 |
| **4. 처리 방법** | 일관성, 재사용 | 순서를 고정해서 매번 같은 방식으로 처리 |
| **5. 출력** | 일관성, 재사용 | 형식을 고정해서 예측 가능한 결과 |
| **6. 제약 조건** | 일관성, 협업 | 금지/필수 사항을 명시해서 리스크 제거 |
| **7. 평가 기준** | 개선, 협업 | 체크리스트로 품질 검증, 개선 근거 마련 |
| **8. 기초 지식** | 개선 | 설계 철학을 문서화, 개선 시 참고 |
| **9. 출력 템플릿** | 일관성 | 구체적 예시로 형식을 정확히 전달 |

이 표를 보면 각 요소가 어떤 시스템 속성을 강화하는지 알 수 있습니다. 예를 들어 "입력"은 재사용성과 확장성을 높이고, "출력"은 일관성을 보장하고 재사용 가치를 높이죠.

### 모든 요소가 항상 필요한가?

아닙니다. 구성 요소들은 명확한 기준으로 우선순위가 나뢜니다:

#### ⭐ 필수 5가지: 실행 가능성의 최소 조건

**판단 기준**: 이것 없으면 AI가 작업을 시작조차 할 수 없음

| 요소 | 없으면 발생하는 문제 |
|------|-----------------------|
| 1. 목적 | "뭘 만들어야 하는지" 자체를 모름 → 시작 불가 |
| 2. 역할 | 어떤 관점에서 접근할지 모름 → 방향성 없음 |
| 3. 입력 | 무엇을 받아서 처리해야 할지 모름 → 대상 없음 |
| 4. 처리 방법 | 어떤 절차로 작업할지 모름 → 진행 불가 |
| 5. 출력 | 어떤 형태로 결과를 내야 할지 모름 → 완성 불가 |

**실험**: 필수 요소 하나라도 빠뜨리고 실행해보세요. AI가 반드시 질문합니다.

#### 🔵 권장 2가지: 품질과 일관성의 핵심

**판단 기준**: 없어도 작동하지만, 있으면 실패율이 50% 이상 감소

| 요소 | 효과 |
|------|------|
| 6. 제약 조건 | 금지사항을 명시하지 않으면 자주 위반 (실패율 60% → 10%) |
| 7. 평가 기준 | 품질 검증이 없으면 결과물이 들 ucb49날쭉 (불량률 40% → 5%) |

**경험 법칙**: v1.0에서는 생략해도 되지만, 반복 사용하면 반드시 추가하게 됨

#### ⚪ 선택 2가지: 장기 효율과 유지보수

**판단 기준**: 초기에는 불필요하지만, 반복 사용하면 효율이 누적됨

| 요소 | 누적 효과 |
|------|----------|
| 8. 기초 지식 | 설계 의도를 문서화하여 개선 시 시행착오 90% 감소 |
| 9. 출력 템플릿 | 구체적 예시로 팀원 설명 시간 90% 단축 |

**적용 시점**: 
- 기초 지식: 3번째 버전(v3.0) 업데이트할 때
- 출력 템플릿: 다른 팀원이 사용하기 시작할 때

---

### 단계별 작성 전략

실제로 인스트럭션을 만들 때는 이런 순서로 접근하세요:

```
v1.0 (첫 사용)
→ 필수 5가지만 채우기 (30분 투자)
→ 일단 실행해보기

v1.1-1.5 (1-2주 사용)
→ 문제 발견할 때마다 수정
→ 제약 조건이나 평가 기준 추가 고려

v2.0 (안정화)
→ 권장 2가지 추가 (10분 투자)
→ 이제 안정적으로 반복 사용

v3.0+ (성숙)
→ 팀 확산 시점에 선택 2가지 추가 (20분 투자)
→ 완전한 시스템으로 완성
```

처음부터 9가지를 모두 채우려고 하지 마세요. 필수 5가지로 시작해서 필요에 따라 점진적으로 추가하는 것이 현실적입니다.

## 3.3 표준 인스트럭션 템플릿

이제 이론을 실전으로 옮길 차례입니다. 9가지 구성 요소를 실제로 어떻게 작성하는지 템플릿으로 만들어보겠습니다.

이 템플릿은 여러분이 바로 복사해서 사용할 수 있도록 설계되었습니다. 각 항목마다 간단한 설명과 예시를 함께 제공하니, 상황에 맞게 수정하시면 됩니다.

### 각 구성 요소 설명

# [인스트럭션 제목]

#### 1. 목적 (Purpose) ⭐ 필수

**무엇을 작성하나요?**  
이 인스트럭션으로 달성하려는 최종 목표를 한 문장으로 작성합니다.

**예시:**  
주간 판매 추이를 분석하여 임원 의사결정을 지원합니다.

**왜 필요한가요?**  
팀 전체가 같은 목표를 공유하고, 개선 방향을 명확히 하기 위해서입니다.

---

#### 2. 역할 (Role) ⭐ 필수

**무엇을 작성하나요?**  
AI가 수행할 전문가 역할을 지정합니다.

**예시:**  
10년차 데이터 분석가 (비즈니스 인사이트 중심)

**왜 필요한가요?**  
답변의 톤과 깊이를 일관되게 유지하기 위해서입니다.

---

#### 3. 입력 (Input) ⭐ 필수

**무엇을 작성하나요?**  
AI가 받아서 처리할 데이터의 형식, 구조, 출처를 설명합니다.

**예시:**
- 파일: sales_{YYYYMMDD}.csv
- 필수 필드: date, product_id, amount, region
- 기간: 최근 7일

**왜 필요한가요?**  
파라미터화해서 재사용 가능하게 만들고, 다른 데이터에도 확장 가능하게 하기 위해서입니다.

---

**💡 파라미터 전달 방법**

입력에 `{YYYYMMDD}` 같은 파라미터가 있을 때, 실제 실행 시 값을 전달하는 방법은 두 가지입니다:

**방법 1: 대화형 (AI가 질문)**
```markdown
## 3. 입력
- 파일: sales_{YYYYMMDD}.csv
- 기간: {YYYYMM}
- 지역: {REGION}

**AI 지침**: 파라미터가 `{}`로 남아있으면 사용자에게 먼저 질문하세요.
``` 

→ 실행 시: AI가 "YYYYMMDD 값을 알려주세요"라고 먼저 질문합니다.  
→ 장점: 사용자가 편함 (파일 그대로 붙여넣기)  
→ 단점: 매번 대화 필요 (자동화 어려움)

**방법 2: 직접 입력형 (사용자가 미리 채움)**
```markdown
## 3. 입력

**실행 전 필수**: 아래 빈칸을 실제 값으로 채우세요

- 파일: sales___________.csv  ← 날짜 입력 (예: 20241106)
- 기간: ________  ← 연월 입력 (예: 202411)
- 지역: _____  ← 지역 코드 (예: SEL)
```

→ 실행 시: 사용자가 빈칸을 채워서 AI에게 전달합니다.  
→ 장점: 빠르고 자동화 가능  
→ 단점: 사용자가 직접 수정 필요

**권장 방식: 두 가지 옵션 모두 제공**
```markdown
## 3. 입력

### 방법 A: 직접 입력 (빠른 실행)
아래 빈칸을 채워서 AI에게 전달:
- 파일: sales___________.csv (예: sales_20241106.csv)
- 기간: ________ (예: 202411)

### 방법 B: 대화형 (초보자용)
아래를 그대로 전달하면 AI가 질문:
- 파일: sales_{YYYYMMDD}.csv
- 기간: {YYYYMM}

**AI 지침**: `{}`가 남아있으면 사용자에게 값을 물어보세요.
```

---

#### 4. 처리 방법 (Process) ⭐ 필수

**무엇을 작성하나요?**  
작업을 수행할 구체적인 절차를 순서대로 나열합니다.

**예시:**
1. 제품별 총 판매액 집계 (SUM by product_id)
2. 전주 대비 증감률 계산 ((이번주-지난주)/지난주*100)
3. 판매액 기준 내림차순 정렬
4. 상위 3개 제품 선정

**왜 필요한가요?**  
순서를 고정해서 일관성을 보장하고, 결과를 재현 가능하게 만들기 위해서입니다.

---

#### 5. 출력 (Output) ⭐ 필수

**무엇을 작성하나요?**  
결과물의 형식과 내용을 구체적으로 명시합니다.

**예시:**
- 형식: 마크다운 표
- 컬럼: product_id | amount | change_rate
- 길이: 500자 (±50자)
- 구조: 요약(3줄) / 분석 표 / 제안(2줄)

**왜 필요한가요?**  
형식을 고정해서 일관성을 보장하고, 결과를 예측 가능하게 만들기 위해서입니다.

---

#### 6. 제약 조건 (Constraints) 🔵 권장

**무엇을 작성하나요?**  
반드시 지켜야 할 한계나 피해야 할 조건을 설정합니다.

**예시:**
- 필수: 500자 (±50자) 이내
- 금지: 전문 용어 사용 금지 (임원 대상)
- 금지: 개인정보 포함 금지
- 금지: 추측성 표현 금지 (데이터 기반만)

**왜 필요한가요?**  
일관성을 보장하고, 리스크를 제거하며, 협업 시 공통 기준을 만들기 위해서입니다.

---

#### 7. 평가 기준 (Evaluation) 🔵 권장

**무엇을 작성하나요?**  
결과물의 품질을 판단할 수 있는 구체적이고 검증 가능한 기준을 체크리스트로 제공합니다.

**예시:**
- [ ] 3개 제품이 포함되었는가
- [ ] 증감률이 % 표시되었는가
- [ ] 500자 (±50자) 범위 내인가
- [ ] 전문 용어가 없는가
- [ ] 마크다운 표 형식인가

**왜 필요한가요?**  
품질을 검증하고, 개선 근거를 마련하며, 협업 시 공통 기준을 만들기 위해서입니다.

---

#### 8. 기초 지식 (Principles) ⚪ 선택

**무엇을 작성하나요?**  
작업 전반에 걸쳐 AI가 항상 참고해야 할 기본 원칙이나 배경지식을 정의합니다.

**예시:**
- 상관관계를 인과관계로 해석하지 말 것

## 3.4 실전 예시: 프롬프트를 시스템으로

이론과 템플릿만으로는 와닿지 않을 수 있습니다. 실제로 채팅창의 즉흥적 프롬프트를 체계적인 인스트럭션으로 바꾸는 과정을 함께 살펴보겠습니다.

### Before: 즉흥적 사용 (채팅창)

다음은 여러분이 채팅창에서 흔히 쓸 법한 프롬프트입니다:

```
"주간 판매 데이터 분석해서 보고서 만들어줘"
```

간단하고 편하죠? 하지만 5가지 시스템 속성 관점에서 보면:

- ❌ **재사용**: 다음 주에 또 새로 작성해야 함
- ❌ **일관성**: 매번 다른 형식으로 나옴 (5페이지? 2페이지? 표? 불릿?)
- ❌ **개선**: 뭘 바꿔야 할지, 이전에 뭘 바꿨는지 모름
- ❌ **협업**: 동료가 이 한 줄만 보고 같은 결과를 만들 수 없음
- ❌ **확장**: 다른 기간, 다른 지역 데이터에 적용 불가

시스템으로서는 0점입니다.

### After: 체계적 사용 (파일)

이제 같은 내용을 9가지 구성 요소로 체계화해보겠습니다. 이 파일을 복사해서 바로 사용하실 수 있습니다:

```markdown
📄 weekly-sales-report.instruction.md

# 주간 판매 분석 보고서

## 변경 이력
- v1.0 (2024-01): 초기 버전
- v2.0 (2024-02): 증감률 컬럼 추가 (CFO 요청)
- v2.1 (2024-03): 500자 제한 추가 (보고서 표준화)

## 1. 목적 ⭐
주간 판매 추이를 분석하여 임원 의사결정을 지원합니다.

## 2. 역할 ⭐
당신은 10년차 데이터 분석가입니다. 비즈니스 인사이트 도출에 집중하며, 
기술적 세부사항보다는 경영진이 이해하고 실행할 수 있는 제안에 초점을 맞춥니다.

## 3. 입력 ⭐
- 파일명: sales_{YYYYMMDD}.csv
- 필수 필드: date, product_id, amount, region
- 데이터 기간: 실행일 기준 최근 7일

## 4. 처리 방법 ⭐
다음 순서로 분석을 수행합니다:

1. 데이터 검증: 필수 필드가 모두 있는지 확인
2. 제품별 집계: 제품별 총 판매액 계산 (SUM by product_id)
3. 증감률 계산: 전주 대비 증감률 산출 ((이번주-지난주)/지난주*100)
4. 정렬 및 선정: 판매액 기준 내림차순 정렬 후 상위 3개 제품 선정
5. 인사이트 도출: 증감 추이의 주요 원인 파악

## 5. 출력 ⭐
다음 형식으로 보고서를 작성합니다:

- 형식: 마크다운
- 전체 길이: 500자 (±50자)
- 구조:
  - 핵심 요약 (3줄): 가장 중요한 발견 1가지
  - 분석 표: 상위 3개 제품의 판매액과 증감률
  - 제안 (2줄): 경영진이 취해야 할 액션

## 6. 제약 조건 🔵
다음 규칙을 반드시 준수하세요:

- 전문 용어 사용 금지 (임원이 이해하지 못할 용어 배제)
- 개인정보 포함 금지 (고객명, 담당자명 등)
- 추측성 표현 금지 ("~인 것 같다" 대신 데이터 기반 사실만)
- 500자 한도 엄수 (±50자 허용)

## 7. 평가 기준 🔵
완료 후 다음 항목을 체크하세요:

- [ ] 상위 3개 제품이 모두 포함되었는가
- [ ] 증감률이 % 기호와 함께 표시되었는가
- [ ] 500자 (±50자) 범위 내인가
- [ ] 전문 용어가 없는가
- [ ] 마크다운 표 형식으로 작성되었는가
- [ ] 추측이 아닌 데이터 기반 사실만 포함되었는가

## 8. 기초 지식 ⚪
다음 원칙을 염두에 두고 작업하세요:

- 상관관계와 인과관계를 혼동하지 않기: "A와 B가 함께 증가했다"는 
  "A가 B를 유발했다"를 의미하지 않습니다
- 데이터 기반 판단: 모든 주장은 제공된 데이터에 근거해야 합니다
- 결론 우선 원칙: 임원 보고서는 결론을 먼저 제시하고, 
  상세 내용은 나중에 설명합니다

## 9. 출력 템플릿 ⚪

### 핵심 요약
이번 주 총 판매액은 전주 대비 12% 증가했으며, A001 제품의 급증이 주요 원인입니다.

### 분석 결과
| 제품ID | 판매액 | 증감률 |
|-------|---------|-------|
| A001  | 1,234만 | +45% |
| B002  | 987만  | +8%  |
| C003  | 756만  | -3%  |

### 제안
A001 제품의 재고 확보를 권장합니다. C003 제품의 하락 원인 분석이 필요합니다.
```

### 시스템 속성 검증

이제 5가지 시스템 속성을 다시 체크해보겠습니다:

- ✅ **재사용**: 날짜만 바꿔서 매주 사용 가능
- ✅ **일관성**: 언제 누가 실행해도 같은 형식 (500자, 3개 제품, 표 형식)
- ✅ **개선**: Git으로 변경 이력 추적, 왜 바꿨는지 기록됨
- ✅ **협업**: 신입도 파일만 보고 실행 가능, 팀 전체가 같은 기준 사용
- ✅ **확장**: 다른 CSV 파일, 다른 기간에도 적용 가능

시스템으로서 만점입니다.

### Before와 After의 실질적 차이

표면적으로는 "한 줄"이 "긴 문서"가 된 것처럼 보입니다. 하지만 실제로는:

**Before (한 줄)**:
- 매주 작성 시간: 5분 (기억 더듬고, 비슷하게 작성)
- 결과 품질: 매번 다름
- 팀 협업: 불가능 (각자 다르게 작성)
- 개선: 어려움 (뭘 바꿨는지 모름)
- **10주 총 비용**: 50분 + 품질 불안정 + 협업 불가

**After (문서)**:
- 첫 작성 시간: 30분
- 매주 사용 시간: 30초 (날짜만 변경)
- 결과 품질: 항상 일정
- 팀 협업: 가능 (모두 같은 파일 사용)
- 개선: 쉬움 (버전 관리로 추적)
- **10주 총 비용**: 30분 + (30초 × 10) = 35분 + 품질 안정 + 협업 가능

첫 작성에 30분 투자하면, 10주 후에는 15분을 절약하고, 품질은 안정되며, 팀 전체가 같은 기준으로 작업할 수 있습니다.

- 디자인 → 와이어프레임 피드백 / 컬러 팔레트 제안 / 레이아웃 검토
- 영업 → 제안서 작성 / 미팅 요약 / 후속 이메일
- 교육 → 강의 자료 생성 / 퀴즈 문제 출제 / 피드백 작성

3장 끝의 실습 과제에서 여러분의 도메인으로 직접 만들어보세요!

## 3.5 시스템 품질 평가 (5점 척도)

인스트럭션의 품질을 정량적으로 측정하고 개선 우선순위를 판단할 수 있는 평가 시스템입니다. Yes/No 방식과 달리, 부분적 달성 상태를 정확히 진단하고 어디서부터 개선해야 할지 명확히 알 수 있습니다.

### 평가 방법

각 시스템 속성은 **5점 만점**으로 평가하며, 전체 합산 점수로 인스트럭션의 성숙도를 판단합니다.

**전체 점수 해석** (총 25점 만점):
- **20-25점**: 우수 - 팀 전체 공유 및 장기 운영 가능
- **15-19점**: 실용적 - 개인/소규모 팀 사용 적합
- **10-14점**: 최소 - 작동은 하지만 개선 필요
- **0-9점**: 불충분 - 시스템이라 보기 어려움, 재작성 권장

---

### 1. 재사용 가능성 (5점 만점)

#### 평가 항목

- [ ] **하드코딩 제거 (2점)**: 날짜, 이름, 제품명 등 특정 값이 파라미터로 대체되었는가?
  - 2점: 모든 가변 값이 파라미터화됨
  - 1점: 일부 하드코딩 존재 (3개 미만)
  - 0점: 다수의 하드코딩 또는 파라미터화 없음

- [ ] **입력 일반화 (2점)**: 입력 형식이 다양한 데이터에 적용 가능하도록 정의되었는가?
  - 2점: 명확한 입력 규격, 다른 데이터에 즉시 적용 가능
  - 1점: 입력 정의는 있으나 특정 데이터에 종속적
  - 0점: 입력이 특정 파일/상황에만 작동

- [ ] **수정 불필요 (1점)**: 다음 실행 시 파일 수정 없이 재사용 가능한가?
  - 1점: 입력 값만 변경하면 즉시 재사용
  - 0점: 매번 파일 내용 수정 필요

**점수**: __/5점

**개선 우선순위**:
- 0-2점: 🔴 즉시 개선 필요 (재사용 불가능)
- 3-4점: 🟡 점진적 개선 (부분적 재사용)
- 5점: 🟢 유지 (완전한 재사용)

**개선 팁**:
- 구체적 값 → 변수명으로 변경 (예: "2024-03" → "{YYYYMM}")
- 특정 파일명 → 패턴으로 변경 (예: "sales.csv" → "sales_{date}.csv")
- 하나의 케이스 → 여러 케이스로 확장 (예: "서울" → "지역 코드: SEL/BUS/...")

---

### 2. 일관성 보장 (5점 만점)

#### 평가 항목

- [ ] **모호성 제거 (2점)**: "적절히", "충분히", "대략" 같은 모호한 표현이 제거되었는가?
  - 2점: 모든 표현이 구체적 수치/기준으로 명시됨
  - 1점: 일부 모호한 표현 존재 (3개 미만)
  - 0점: 다수의 모호한 표현 또는 기준 없음

- [ ] **출력 형식 명시 (2점)**: 출력 형식이 구체적으로 정의되었는가?
  - 2점: 형식, 길이, 구조 모두 수치로 명시
  - 1점: 형식만 명시, 길이/구조 모호
  - 0점: 출력 형식 정의 없음

- [ ] **재현 가능성 (1점)**: 같은 입력이면 같은 형식의 결과가 나오는가?
  - 1점: 5회 테스트 시 모두 동일 형식
  - 0점: 테스트 시 형식이 달라짐

**점수**: __/5점

**개선 우선순위**:
- 0-2점: 🔴 즉시 개선 필요 (결과 예측 불가)
- 3-4점: 🟡 점진적 개선 (대체로 일관됨)
- 5점: 🟢 유지 (완벽한 일관성)

**개선 팁**:
- "적절히" → "500자 (±50자)" 같은 구체적 수치
- "간단히" → "3문장 이내" 같은 명확한 기준
- "표로" → "마크다운 표, 3개 컬럼" 같은 상세 명세

### 3. 개선 가능성 (5점 만점)

#### 평가 항목

- [ ] **변경 이력 (2점)**: 버전별 변경 내용과 이유가 기록되어 있는가?
  - 2점: 날짜, 변경 내용, 이유 모두 기록
  - 1점: 변경 내용만 있고 이유 없음
  - 0점: 변경 이력 없음

  - 1점: 일부 주석으로 설명
  - 0점: 설계 의도 설명 없음

- [ ] **품질 검증 (1점)**: 평가 기준으로 결과물 품질을 검증할 수 있는가?
  - 1점: 체크리스트 형태의 평가 기준 존재
  - 0점: 평가 기준 없음

**점수**: __/5점

**개선 우선순위**:
- 0-2점: 🔴 즉시 개선 필요 (개선 방향 불명확)
- 3-4점: 🟡 점진적 개선 (일부 추적 가능)

**개선 팁**:

- "왜"를 항상 함께 기록 (예: "v2.0: 증감률 추가 (CFO 요청)")

- ⚪ **선택 2개** (장기 효율과 유지보수): 기초지식, 출력 템플릿

### 실습 과제

**과제 1: 시스템 속성 진단**

자신이 자주 사용하는 프롬프트 하나를 선택하여 5가지 시스템 속성 관점에서 분석하세요:

현재 상태:
- [ ] 재사용 가능: 하드코딩된 값이 없고 파라미터화되어 있는가?
- [ ] 일관성 보장: 모호한 표현이 없고 구체적 기준이 있는가?
- [ ] 개선 가능: 변경 이력을 추적할 수 있는가?
- [ ] 협업 가능: 구두 설명 없이 팀원이 사용할 수 있는가?
- [ ] 확장 가능: 다른 데이터나 상황에도 적용 가능한가?

각 속성별로:
- 현재 점수: 0-5점
- 부족한 이유: 구체적으로 무엇이 문제인가?
- 개선 방법: 어떻게 고치면 되는가?

**과제 2: 9가지 구성 요소 템플릿 작성**

자신의 업무에서 반복되는 작업 하나를 선택하여 9가지 구성 요소 템플릿을 채우세요:

최소 요구사항:
- 필수 5가지는 반드시 작성
- 권장 2가지 중 최소 1개 작성
- 각 요소가 어떤 시스템 속성을 만족시키는지 표시
- 실제로 사용할 수 있는 수준으로 구체적으로 작성

**과제 3: Before/After 비교**

과제 2에서 만든 인스트럭션을 채팅창 버전과 비교하세요:

**1. Before (채팅창 버전)**: 
같은 작업을 채팅창에서 어떻게 했는지 작성하세요.

**2. After (파일 버전)**: 
과제 2에서 만든 9가지 요소 버전

**3. 개선 효과**:
각 시스템 속성별로 어떻게 개선되었는지 간단히 설명하세요.

| 시스템 속성 | Before (채팅) | After (파일) |
|-----------|--------------|-------------|
| 재사용 가능 | 예: 매번 새로 작성 | 예: 입력 데이터만 변경 |
| 일관성 보장 | 예: 형식이 매번 다름 | 예: 항상 같은 형식 |
| 개선 가능 | 예: 변경 추적 불가 | 예: Git으로 이력 관리 |
| 협업 가능 | 예: 구두 설명 필요 | 예: 파일만으로 실행 |
| 확장 가능 | 예: 특정 데이터에만 | 예: 다른 데이터에도 적용 |

**4. 투자 대비 효과 (선택사항)**:
시간이 허락한다면 계산해보세요.
- 첫 작성 시간: ___ 분
- 주당 사용 횟수: ___ 회  
- 매회 절약 시간: ___ 분
- 10주 후 총 절감: ___ 분
