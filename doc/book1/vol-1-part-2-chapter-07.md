# 7장. 실전 패턴: 단일 에이전트 설계

**Part 2: 설계 원칙과 구성 요소**

**목적:** 한 명의 AI 전문가가 독립적으로 작업을 완료하는 '단일 에이전트' 패턴을 세 가지 복잡도 수준에 맞춰 학습하고, 이 과정에서 사용자가 어떻게 '메타 에이전트'의 역할을 수행하는지 이해합니다.

### 이 장에서 배우는 것
- 간단한 반복 작업부터 복잡한 분석 작업까지, 단일 에이전트의 복잡도를 조절하는 방법을 익힙니다.
- 단일 에이전트 시스템에서 사용자가 수행하는 '메타 에이전트'의 역할을 이해합니다.
- 이 장의 예제들을 템플릿으로 활용하여 자신의 문제에 맞는 인스트럭션을 설계할 수 있습니다.

---

## 7장 전체 사례 맵

이 장(7장)은 에이전트 수와 작업 복잡도에 따라 3가지 실전 사례를 제공합니다.

| 복잡도 | 사례 |
|---|---|
| **간단/일상**<br>(낮은 위험, 단순 작업) | 이메일 요약 |
| **표준/전문**<br>(중간 복잡도, 전문 지식) | 마케팅 문구 생성 |
| **복잡/중요**<br>(높은 위험, 법적 책임) | 회의록 할 일 추출 |

**참고:** 이 사례들은 4장의 '상황별 필수 원칙 매트릭스'와 연계되어 있습니다. 각 사례는 해당 복잡도에서 필수적으로 적용해야 할 메타 원칙들을 실전에 적용한 예시입니다.

---

## 들어가며: 나만의 AI 전문가 만들기

'단일 에이전트'는 가장 기본적이면서도 가장 많이 사용되는 패턴입니다. 이는 특정 과업을 맡기기 위해 전문가 한 명, 즉 **'워커 에이전트'**를 고용하는 것과 같습니다.

### 메타 에이전트의 역할

이 패턴에서 **메타 에이전트의 역할은 바로 '사용자' 자신**이 수행합니다.
1.  **목표 설정:** "이메일을 요약해야겠다" 또는 "마케팅 문구가 필요해" 와 같이 해결할 문제를 정의합니다.
2.  **에이전트 생성 및 지시:** 머릿속으로 혹은 텍스트 편집기에서 특정 역할('전문 요약 비서')과 규칙을 가진 워커 에이전트의 인스트럭션을 **설계하고 생성**합니다.
3.  **실행 및 감독:** 생성한 인스트럭션을 AI에 전달하여 워커 에이전트를 실행하고, 그 결과물을 직접 평가합니다.

즉, 단일 에이전트를 활용하는 것은 우리가 무의식적으로 메타 에이전트의 역할을 수행하며 AI 워커와 1:1로 협업하는 과정이라고 이해할 수 있습니다.

## 7.1 [사례 1] 간단/일상: 이메일 요약

- **상황:** 매일 수신하는 이메일 뉴스레터를 빠르게 훑어보기 위해 3줄 요약이 필요하다.
- **메타 에이전트(사용자)의 목표:** "뉴스레터의 핵심만 빠르게 파악할 워커를 만들자."

### 최종 인스트럭션 예시 (워커 에이전트용)
```markdown
# 역할: 전문 요약 비서
# 목표: 주어진 이메일 본문을 3개의 불릿 포인트로 요약한다.
# 출력 규칙
- 요약은 반드시 3개의 불릿(-)으로 구성되어야 한다.
- 각 불릿은 완전한 한글 문장으로 끝나야 한다.
- 가장 중요한 결론이나 액션 아이템을 첫 번째 불릿에 위치시킨다.
```

## 7.2 [사례 2] 표준/전문: 마케팅 문구 생성

- **상황:** 신제품 출시에 맞춰, 제품의 핵심 특징을 담은 마케팅 캐치프레이즈 5개를 제안받고 싶다.
- **메타 에이전트(사용자)의 목표:** "우리 제품의 핵심 가치를 잘 이해하고 매력적인 문구를 만들어낼 카피라이터 워커를 만들자."

### 최종 인스트럭션 예시 (워커 에이전트용)
```markdown
# 역할: 10년차 시니어 카피라이터
# 목표: 주어진 `[제품 정보]`를 바탕으로, 신제품의 매력을 어필할 수 있는 마케팅 캐치프레이즈 초안 5개를 생성한다.
# 컨텍스트
- 대상 고객: 최신 기술에 관심이 많은 20-30대 남성
- 브랜드 톤앤매너: 혁신적, 자신감, 유머러스함
# 출력 형식
- 5개의 캐치프레이즈를 번호 목록으로 제시한다.
- 각 캐치프레이즈 아래에는 어떤 특징을 강조했는지 1줄로 간략한 설명을 덧붙인다.
# 제품 정보
- 제품명: '퀀텀 AI 글래스'
- 핵심 특징: 실시간 통역, AR 내비게이션, 세련된 디자인
```

## 7.3 [사례 3] 복잡/중요: 회의록 기반 할 일 추출

- **상황:** 긴 회의가 끝난 후, 회의록 텍스트에서 할 일(Action Item) 목록만 정확히 뽑아내고 싶다.
- **메타 에이전트(사용자)의 목표:** "회의록의 미묘한 맥락을 이해하고, 할 일을 정확하게 구조화하여 추출하는 꼼꼼한 PM 워커를 만들자."

### 최종 인스트럭션 예시 (워커 에이전트용)
```markdown
# 역할: 꼼꼼한 프로젝트 관리자 (PM)
# 목표: 주어진 회의록 텍스트를 분석하여, 모든 '할 일(Action Item)'을 식별하고 구조화된 JSON 형식으로 추출한다.
# 처리 방법
1. 회의록 전체를 읽고, 명시적으로 할당된 모든 작업을 식별한다.
2. 각 작업 항목에서 '담당자', '업무 내용', '마감 기한' 세 가지 요소를 추출한다.
3. 아래 `[출력 형식]`에 정의된 JSON 스키마에 맞춰 보고서를 생성한다.
# 제약 조건
- 만약 '담당자', '업무 내용', '마감 기한' 중 하나라도 불분명하거나 누락된 항목이 있다면, 자의적으로 추측하지 말고 해당 항목의 값을 `null`로 설정하고 `needs_clarification` 필드를 `true`로 설정하라.
# 출력 형식
- 아래 JSON 스키마를 정확히 준수하는 단일 JSON 객체를 생성한다.

[action_items.schema.json]
```json
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "title": "Action Item List from Meeting Minutes",
  "type": "object",
  "properties": {
    "action_items": {
      "type": "array",
      "items": {
        "type": "object",
        "properties": {
          "assignee": { "type": ["string", "null"] },
          "task_description": { "type": "string" },
          "due_date": { "type": ["string", "null"], "format": "date" },
          "needs_clarification": { "type": "boolean" }
        },
        "required": ["task_description", "needs_clarification"]
      }
    }
  },
  "required": ["action_items"]
}
```

### 🎯 핵심 개념 복습 (5가지)

1권을 관통하는 가장 중요한 5가지 개념:

1. **프롬프트 < 인스트럭션**: 일회성 질문보다 재사용 가능한 업무 매뉴얼이 더 강력합니다
2. **명확성 = 품질**: 모호함을 제거할수록 AI는 정확한 결과를 냅니다
3. **단일 책임 원칙**: 하나의 에이전트는 하나의 명확한 역할만 수행해야 합니다
4. **워크플로우 패턴**: 파이프라인, 생성-검증, 라우팅으로 복잡한 작업을 해결합니다
5. **측정하고 개선하기**: 평가 지표 없이는 개선도 없습니다

- 매일 사용하며 개선점 기록

**Step 2: 워크플로우 적용** (2-4주)
- 여러 단계가 필요한 작업 선정 (예: 보고서 작성, 콘텐츠 생성)
- 7장의 워크플로우 패턴 적용 (파이프라인 또는 생성-검증)
- 9장의 평가 방법으로 성능 측정

**Step 3: 시스템화** (1-2개월)
- 팀에서 자주 하는 업무 3-5개 선정
- 10장의 복잡도별 패턴 참고하여 표준 인스트럭션 세트 구축
- 팀원들과 공유하고 피드백 수집

**💡 실무 적용 팁**:
- 처음부터 완벽하려고 하지 마세요. 80% 수준에서 시작하여 점진적으로 개선하세요.
- 작은 성공을 축적하세요. 간단한 작업부터 자동화하여 신뢰를 쌓으세요.
- 문서화하세요. 잘 작동하는 인스트럭션은 팀 자산이 됩니다.

---

### 🔍 자기 점검 체크리스트

다음 질문에 "예"라고 답할 수 있다면 1권을 성공적으로 완료하신 것입니다:

**기본 이해도**
- [ ] 프롬프트와 인스트럭션의 차이를 설명할 수 있나요?
- [ ] 좋은 인스트럭션의 4가지 특성을 기억하고 있나요?
- [ ] 6가지 질문 유형 중 3가지 이상을 실제로 사용해봤나요?

**설계 능력**
- [ ] 단일 책임 원칙에 따라 에이전트를 설계할 수 있나요?
- [ ] 입력/출력을 명확하게 명세할 수 있나요?
- [ ] 메타 원칙을 2-3가지 이상 적용할 수 있나요?

**실전 적용**
- [ ] 3가지 워크플로우 패턴 중 하나를 사용해봤나요?
- [ ] 평가 지표를 설정하고 결과를 측정해봤나요?
- [ ] 자신의 업무에 맞는 인스트럭션을 작성해봤나요?

**만약 체크가 안 된 항목이 있다면**: 해당 장으로 돌아가서 실습 체크리스트를 다시 수행해보세요.

### ⚠️ 다음 단계가 필요한 상황

다음과 같은 상황에 직면했다면 **2권**이 필요합니다:

**복잡도 증가**
- ⚠️ 단일 에이전트만으로는 해결하기 어려운 복잡한 문제에 직면했을 때
- ⚠️ 여러 단계의 검토와 승인이 필요한 작업이 있을 때
- ⚠️ 다양한 전문 지식이 필요한 복합적인 작업이 있을 때

**협업 필요성**
- ⚠️ 여러 AI 에이전트를 협업시켜야 할 필요성을 느낄 때
- ⚠️ 에이전트 간 작업 결과를 공유하고 누적해야 할 때
- ⚠️ 파일 기반으로 작업을 관리하고 추적해야 할 때

**조직 확장**
- ⚠️ 팀 또는 조직 전체의 AI 시스템을 설계해야 할 때
- ⚠️ 관리자-실무자 구조의 계층적 시스템이 필요할 때
- ⚠️ 대규모 에이전트 오케스트레이션이 필요할 때

**→ 이런 상황이라면 2권에서 다루는 '다중 에이전트 협업'과 '조직 설계'가 필요합니다.**

### 📘 2권 예고: 조직 수준의 AI 설계

1권에서는 '한 명의 AI 전문가'를 효과적으로 활용하는 방법을 배웠습니다. 하지만 복잡한 비즈니스 문제는 종종 **여러 전문가의 협업**이 필요합니다.

2권에서는 **조직과 팀 수준**에서 **다중 AI 에이전트 협업 시스템**을 설계하는 방법을 다룹니다.

#### **2권의 주요 주제**

**Part 4 (계속): 조직 설계 아키텍처**
- **11장**: 파일 기반 협업 시스템 - 여러 에이전트가 파일을 통해 작업을 공유하고 협업하는 방법
- **12장**: 계층적 협업 아키텍처 - 관리자-실무자 구조의 에이전트 시스템 설계
- **13장**: 단위 조직 설계 - 기능별 에이전트 팀 구성 및 관리
- **14장**: 복합 조직 설계 - 대규모 에이전트 오케스트레이션 및 조정

**Part 5: 진화와 관리**
- **15장**: 인간-AI 협업 조직의 진화 - 조직 변화와 AI 시스템의 지속적 진화 관리

#### **실전 사례 (2권)**
- 소셜 미디어 콘텐츠 자동 발행 시스템
- 신제품 출시 캠페인 전체 자동화
- 조직 유형별 AI 협업 패턴 (Startup, Enterprise, Non-profit)

#### **1권 vs 2권 비교**

| 구분 | 1권 (단일 에이전트) | 2권 (다중 에이전트 조직) |
|------|---------------------|-------------------------|
| **대상** | 개인, 소규모 팀 | 조직, 기업 |
| **범위** | 한 가지 작업 자동화 | 여러 작업의 협업 시스템 |
| **복잡도** | ★☆☆~★★☆ | ★★★ |
| **핵심 패턴** | 단일 에이전트, 기본 워크플로우 | 파일 기반 협업, 계층 구조 |
| **의사결정** | 단순 (단일 에이전트) | 복잡 (다중 에이전트 조정) |
| **예제** | 이메일 요약, 마케팅 문구 | 콘텐츠 발행 시스템, 캠페인 자동화 |
